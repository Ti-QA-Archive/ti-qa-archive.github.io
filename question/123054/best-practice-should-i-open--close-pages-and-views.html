Best Practice: Should I open / close pages and views?

Hi All,
Bit of a newbie code structure question but I hope you can help.
When moving between .js pages/windows, should I close one before opening the next?

Example: I have 3 pages/windows in my app:
Home (app.js)
About (info.js)
Game (play.js)

Imagine that the user is on the Game page (play.js) and in that page they click on the About button (which loads info.js). Should I first try close Game.js then open About.js or the otherway around, or do I even need to do that?

I'm assuming I need to do something to stop the views & windows piling up and draining memory?

Thanks,
Lee

  • Accepted Answer

    I assume that your are using a TabGroup with child windows defined in separate js files, as in the Kitchensink app. If that is the case, just three root windows as you describe should not pose a memory management issue that Titanium cannot handle automatically.

    The memory management stuff becomes more crucial when you have, lets say, a window, with controls which open another window, and perhaps you are updating and loading new views, or tableViews. All of this UI objects consume memory, and they need to be closed to release that memory at some point. In any case, when you close a window, all views in the window are released from memory.

    There are UI objects, such as Tabs or Navigation groups, which do some window management for you. When you open new windows in a Tab or Navigation group, you have the 'back' button displaying automatically. When you click on the button, the window is closed for you, releasing its memory footprint.

    But if you open windows and views manually, not using navigation groups or tabs, you have to be more careful and close them when no longer needed. Specially Tableviews can take a lot of memory…

  • Every time you are using:
    For win0

    var win1=Ti.UI.createWindow({url:'win1.js'};
win.open();

    You are opening a window in its own execution context. This means that the 'originating' and 'created' windows do not share namespace and do not have access to each other. And , in your example, you'll have 4 windows in the stack.

    So a a button opening win0 from win1, is creating a win0 in a new execution context, different from the original win0, even if both 'win0' would share a 'win0.js'. This is not very advisable anyway, and you definitely would have to handle the closing of this windows, but it would probably become very messy.

    If you wouldn't be creating windows with a file.js url (which creates a new execution context), then the approach would be more sound. I am not sure about this, but I think you still need to close the previous instance of win0, because I think it doesn't get garbage collected.

    Anyway, if you are accessing the same windows, within the same execution context you could always show them and hide them instead of creating them, while keeping object references.
