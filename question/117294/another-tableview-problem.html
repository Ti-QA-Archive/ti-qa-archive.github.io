  • I mostly see this issue when I test my apps on the iPhone 3G. 3GS and higher are much faster.

    Are you doing anything out of the ordinary when you create your sections, rows, etc? How many rows are you trying to display? Some of my tableviews are have over 50 rows and include custom section header views, along with custom rows containing image views and a bunch of styled labels. I have found that the time lag is spent in the loop that creates the sections and rows. This is because the loop is executing in the Javascript runtime which is much slower than native Objective-C. So the more sections/rows you have the longer it's going to take to create the tableview data for display.

    One thing I have done to make this appear more friendly to the user is to continue to show my loading indicator until all the sections and rows have been created, and then after calling setData I remove the loading indicator. If you want to support iPhone 3G this is almost a must have, otherwise to the user things will appear to come to a crawl.

    Best,

    Brian

  • I had the exact same issue, Scott. I noticed that the rows in my tables were loading very slow. To begin with I had 4 images and 2 labels attached without a className.

    Class Name: I added a className and noticed 50% speed increase.

    Simple Table: I began reading the wiki.appcelerator.org guide about performance tips and they suggested using "simple tables". I did this and noticed at least a 75% speed increase.

    Balance: I compromised speed with design by using a "standard table" with 2 labels and of course a className, and the speed was still close to 70% faster or better. I suspect the use of images slows everything down. See my post here for more details.
