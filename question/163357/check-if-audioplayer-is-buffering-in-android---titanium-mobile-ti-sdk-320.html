Check if audioPlayer is buffering in Android - Titanium Mobile (Ti SDK 3.2.0)

Hello Everyone,

I'm currently developing a titanium mobile application. I am working on an implementation of a radio stream using the Audio Player of Titanium. Is there any way to check if the audio player is buffering the stream? Sometimes it occurs.

I know from the following question on stackoverflow that the change event is not fired when buffering, also checked it myself and it doesn't
(http://stackoverflow.com/questions/12847021/get-audioplayer-state-on-titanium-mobile-for-android

When I'm debugging on a real device it shows a warning though that indicates that the Android MediaPlayer is buffering:

  • [WARN] : MediaPlayer: info/warning (703, 343)
  • [WARN] : MediaPlayer: info/warning (701, 0)
  • [INFO] : MediaPlayer: Info (703,343)
  • [INFO] : MediaPlayer: Info (701,0)
  • [WARN] : MediaPlayer: info/warning (702, 0)
  • [INFO] : MediaPlayer: Info (702,0)

Is there any way to catch the 'buffering' event?

Yours sincerely,
Patrick van Vuuren

    • This does not work unfortunately, because the change event is not fired when the media player (android) is buffering/change network. Tested it with the following code:

      this.audioPlayer = Ti.Media.createAudioPlayer({ 
     url:Config.radioUrl,
     allowBackground:true, // For iPhone specify background in tiapp.xml
     preload:true
})
this.audioPlayer.addEventListener("change",function(e) {
     Ti.API.info(e.description);        
});

      Outputs:

      [INFO] :   ----DEVELOPERS REPORT: starting
[INFO] :   MediaPlayer: Dont send intent. msg.arg1 = 0, msg.arg2 = 0
[INFO] :   ----DEVELOPERS REPORT: initialized
[INFO] :   ----DEVELOPERS REPORT: playing

      (Below happened like 15 seconds later when it was buffering)

      [WARN] :   MediaPlayer: info/warning (703, 433)
[INFO] :   MediaPlayer: Info (703,433)
[WARN] :   MediaPlayer: info/warning (701, 0)
[INFO] :   MediaPlayer: Info (701,0)
[WARN] :   MediaPlayer: info/warning (702, 0)
[INFO] :   MediaPlayer: Info (702,0)

      As you see, no change event is fired.
