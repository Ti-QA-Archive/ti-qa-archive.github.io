TableView Height Highly Frustrating

Seems that no combination of height:'auto' will make a TableView appear in a View in a Window.

Only assigning it an absolute height will make it appear.

This and other similar layout issues are so fiddly and pedantic they render the entire layout model of Appcelerator a pain to use. Based on others cries of pain in the forums describing similar issues, this seems to be something endemic within the framework.

Every time I am faced with a layout issue, I am filled with dread, because I cannot solve it with logic, only trial and error and a lot of luck.
Why can't the View and TableView automatically calculate their height, when they are set to 'auto'? Isn't that the whole point of 'auto'?

Is providing an absolute height the only to resolve this issue?

Is using absolute values the only way to achieve layout in Appcelerator?

  • There are some UI controls that don't play ok with a parent that has the layout set, but my understanding is that is normal not to. Let me explain:

    In your case the tableview has in fact 2 components:
    one is the tableview itself that holds inside the sections(or at least one) that are scrollable.
    So there is no relation between the height of the table and it's content (from the rendering point of view).

    When you attach the table to a parent , unless it has a fixed height, it will try to fill the whole space it gets, asking for the parent what space is available. When the parent has a vertical layout, it simply cannot reply to the child (the table) because it doesn't know. From here the weird behaviour.

    Now what you could do is to calculate the height of the table and set it depending on the number of rows.
    It't nasty but it's doable.

    The same will behave a scrollview or a scrollable view for horizontal layout.
