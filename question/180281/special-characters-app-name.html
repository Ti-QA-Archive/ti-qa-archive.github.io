  • Start by checking your app ID (com.whatever.foo). App IDs must follow Java package name rules – ASCII letters and numbers and the underscore about all that are permitted. App names should generally follow the same rules. You can use the internationalization functions to specify a different name that's shown to the user. See the internationalizing the app name guide for more details.

  • Hi

    App names should be a simple as possible. Partially due to early restrictions in file systems you are limited to a-z (upper and lower), numbers and underscore. Keep in mind this is only something YOU see, not your users.

    The good news though is you can have whatever you want as your app name from the users perspective and in doing so you also get to take advantage of string files (seriously this is better than it sounds).

    It is good development practice to use a MVC (think Alloy), .As this framework allows you to separates; layout (views), presentation (tss), logic (controllers), models, widgets (reusable combinations) and more.

    This can also be done with the captions, titles and other sections of text that appear within your app. You do not need Alloy for this - it works exactly the same if you take advantage of Alloy or not.

    You do NOT need to internationalise your app to gain the benefits, but you can if you choose to. To solve your problem you only affect the string entry for app title. The default language to support is English so you will be creating an 'en' folder EVEN if you support more than one language. If your app is used by someone on a device where the language setting is anything other than English then your app will provide the language values from the language string files residing in that language folder. If you do not have a supporting language folder - it defaults to the 'en' folder.
