How to set a backgroundimage on Window - so it does not stretch/scale?

Hi

I have a backgroundimage for my app. It is set on a Window tag. This is the Alloy markup:

<Widget id="drawer" src="nl.fokkezb.drawer">
        <Window module="xp.ui" role="leftWindow">
            <Label>I am left</Label>
        </Window>

        <NavigationWindow id="winMain" platform="ios" role="centerWindow">
            <Window class="container" id="winContent">
                <LeftNavButton>
                    <ImageView id="menuIcon" onClick="toggle"></ImageView>
                </LeftNavButton>
                <TitleControl><Label id="title">CatchLog</Label></TitleControl>
                <Label>Content....</Label>
            </Window>
        </NavigationWindow>
    </Widget>

… and the tss:

"Window": {
    statusBarStyle: Ti.UI.iPhone.StatusBar.LIGHT_CONTENT
}
"#winContent": {
    navGroup: true,
    translucent: true,
    navTintColor: '#fff',
    backgroundImage:'images/background.png',
    backgroundColor: 'transparent',
    extendEdges: [
        Ti.UI.EXTEND_EDGE_TOP
    ],
    barImage:'images/transparent.png',
    hideShadow:true,
}

Note: The above also shows how to make a transparent NavigationBar (see more)

To support the above I have created three background.png files:

  • background-586h@2x.png (640 x 1136)
  • background@2x.png (640 x 960)
  • background.png (320 x 480)

On a 3.5 inch display the background looks correct. However, on a 4 inch display the image is stretched to fit the display - this spoiling the logo in the background.

What is the right way to do this? I know I can do it with a screensize/platform check to select the right image - but is that really necessary?

I am on Mac OS 10.9.4, Titanium Studio, build: 3.2.3.201404181442

    You only need to explicity set the 320x480 and the 640x1136 images. The retina version will be substituted automatically when you use the 320x480 image. You can either use the open event of the window to detect the iPhone model and set the background there, or if you upgrade to Alloy 1.4 / sdk 3.3.0 (just released) you can use custom queries to set the value depending on the iPhone model

      I have just tested the images and all rules for retina and retina tall using all combinations of the following;

      • SDKs
        • Ti SDK 3.2.1

      All combinations 24 combinations worked as expected. However not as you may have thought.

      Firstly we should rule some things out; Ti SDK and Retina & Retina Tall have no bearing on the outcome. We know that the Splash works fine. So we are now left with two things;

      • Window Background
      • App Background *

      You do know about the first one, but you may not understand or even know about the second one. I suspect this is where the problem is coming from - I do not mean your lack of knowledge, just the difference.

      Your code shows that you are setting the background image on the window - this seems likely but it will not always work as you imagine it might. To understand this logic, we need to understand what a window is and how much space it has available to it.

      • (1) Full screen app
      • (2) Standard App
      • (3) App with NavigationWIndow
      • (4) App with TabGroup (NavigationWIndow is automatically added)
      • (5) Just for fun
        • (a) iOS 7
        • (b) TitleBar (used on TabGroup [Not TabGroup itself though] or NavigationWindow) set to translucent

      Your window occupies the space available to it AFTER other things have been taken away, so using the numbers above; (any figures mentioned are in dp)

      • (1) Nothing - window occupies full height
      • (2) Status bar height is deducted by 20dp
      • (3) Height of Title/Navigation Bar & Status Bar
      • (4) Height of Title/Navigation Bar, TabGroup bar & Status Bar
      • (5a) iOS7 can occupying the space underneath the Title/Navigation Bar, TabGroup bar and Status Bar
      • (5a) TitleBar is ignored from calculations if this is set

      So if you set a background image you have to know what value has been reduced based on the features your app has at the time the window in question is displayed. So we now come back to the * entry above.

      The App background is a special value - often overlooked, poorly understood and usually rendered useless if used by settings on other elements - again - due to the poor understanding of this feature.

      If your app need a background image to persist across all windows at all times (exceptions can be made) then you do not set a per window background image - you set a per app background image - just one. Whenever you want your users to see this background image you simply set any; Window, NavigationWindow or TabGroup background colour to transparent. Then the app image will be visible. Anything then on your windows will then sit over the top of this image. If you need to have some windows that do not show this as a background - simply set a window (or TabGroup or NavigationWindow) specific background image or colour.

      Using an app background means that the entire device dimensions are covered with the image and nothing else takes away and width or height from its display.

      So how do you set this? (finally he tells us!)

      You add this to the index.js, commonly at the top of the file. You can change the property later on by using the same method again elsewhere - but the app should start with this set before any UI is presented.

      Example;

      Ti.UI.setBackgroundImage("back.png");

$.index.open();

      Try it and report back.

    • As long as you have the entry I suggested above and it correctly points to a known image that exists then that is all you need to set the app background.

      The black background is often the default colour of a window has not been set to transparent to allow the app background to shine through. Think of the app background as your desk and the windows, tabs and other things as sheets of paper on top of the desk. If the sheets of paper are not transparent you cannot see the desk. This is likely (hopefully) what is happening here.

      No changes to the TSS except for the backgroundColor setting added to the app.tss for TabGroup/NavigationWindow/Window as required. Then just make sure the backgroundColor setting for anything else is either transparent of removed (better then the app.tss is on control). FInally make sure no backgroundImage entries persist - they may now be pointing to an image that longer exists in the same location.

      • size; small, normal, large
      • density; ldpi, mdpi, hdpi, xhdpi, xxhdpi, xxxhdpi

        • example-568h@2x~iphone.png
        • example~ipad.png
        • example@2x~ipad.png
      • icons (using iOS real name)
        • Icon.png

    • Yes for iOS each image can be placed in a single folder, then any other density variations are placed inside the same folder

      The files stored in the project including the virtual folder 'iphone';

      • {appname}/app/assets/iphone/images/example.png
      • {appname}/app/assets/iphone/images/example@2x.png
      • {appname}/app/assets/iphone/images/example-568h@2x.png

      as well as these files stored without a virtual folder actually resolve down to this structure any way;

      • {appname}/app/assets/images/example.png
      • {appname}/app/assets/images/example@2x.png
      • {appname}/app/assets/images/example-568h@2x.png

      Once the root prefix of {appname}/app/assets and any virtual folders are removed iphone in this case the image reference in the app is simply /images/{filename}.

      You would NOT have two sets of files - Only one set - choose the structure best suited to your app. Personally I always keep iphone specific images in their virtual folder even if the app is iOS only. This way I have one standard for all apps.

      So I would actually use the iphone virtual folder for all iOS only images and android virtual folder for all android specific images - but any generic non-platform dependant images would go in the non-virtual folder.

      • {appname}/app/assets/iphone/images/only_used_in_ios.png
      • {appname}/app/assets/android/images/only_used_in_android.png
      • {appname}/app/assets/images/not_platform_specific.png

      So iOS gets;

      • /images/only_used_in_ios.png
      • /images/not_platform_specific.png

      And Android gets;

      • /images/only_used_in_android.png
      • /images/not_platform_specific.png

      Back to the code example;

      // controller example;
Ti.UI.setBackgroundImage("/images/example.png");
