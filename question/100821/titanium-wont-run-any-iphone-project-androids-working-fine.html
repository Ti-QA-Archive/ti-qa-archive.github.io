  • Okay, here it goes.
    I don't understand how Ruby's location affects the copyplist from xcode, but I do know how to fix this particular problem.

    In my case, the main problem was:

    [ERROR] error: can't exec '/Developer/Library/Xcode/
Plug-ins/CoreBuildTasks.xcplugin/Contents/
Resources/copyplist' (No such file or directory)

    Copyplist seems to be a tool used by CoreBuildTasks in the building process. I found this out by trying to build the generated xcodeproj file in the build directory. The debugger from xcode singled out this as the only problem, so I figured, all other error codes in the Titanium have to be from the Titanium build scripts. Anyway, since I couldn't find any solution and hardly could find the problem on Titanium forums, Q&A or even by Googling, I copied the debug message from xcode into google instead.

    I came across a couple of blog and forum posts regarding this issue. For instance:
    http://blog.hybridindie.com/2010/10/18/xcode-error-copyplist/
    http://stackoverflow.com/questions/384962/copyplist-failed-with-exit-code-71

    I recognized that, since I just had upgraded Ruby. There's an issue though. The 'default' ruby, which comes with OS X, isn't installed in the same location as the 'new' Ruby. Somehow, Copyplist needs to know where Ruby is, (I don't know why though) and get's confused if it's been replaced by a new ruby.

    The solutions mentioned in the above posts are:

    1) Run the following command in Terminal

    sudo ln -s /opt/local/bin/ruby /usr/bin/ruby

    2) Run the following command in Terminal

    sudo ln -s `which ruby` /usr/bin/ruby

    According to multiple sources (I'm sorry I only listed one, which is the Stack Overflow link) the second (2) solution is the "more universal fix". I'm not sure why, and maybe the sources I used where quoting each other, but that's the one I used and it worked like a charm.

    I didn't use the first solution since it creates a symlink, and that creates a lot more confusion (for instance when installing another new version of Ruby) and I already tried something like that a few days earlier (somehow my Rails only recognizes old Ruby, not the new one) and failed, since it wouldn't create a symlink since 'ruby' exec already existed.

    Anyway, I hope this helps anyone. Do-it-yourself steps:

    1) I get that error

    2) I recently reinstalled, upgraded or downgraded Ruby

    3) I don't know what to do anymore

    4) Use solution 2 ;-)
