Android auto-scaling of images for different screen densities

Hi,

I created two image views like this:

Ti.UI.createImageView({
    width: Ti.UI.SIZE, height: Ti.UI.SIZE,
    image: '/images/green.png'
});
Ti.UI.createImageView({
    width: Ti.UI.SIZE, height: Ti.UI.SIZE,
    image: '/images/blue.png'
});

and put the file green.png into the android/images/res-hdpi directory, and the file blue.png into the android/images/res-mdpi directory. The two image files are identical in size, 100x200 pixels.

I'd expect these two image views to have different sizes when displayed, because the two images should be scaled differently by the auto-scaling mechanism in Android. For example, if I run the app on an MDPI-class device, the file(s) in res-mdpi should remain unscaled while the file(s) in res-hdpi should be scaled down. Similarly, if I run the app on an HDPI-class devices, the file(s) in res-hdpi should remain unscaled while the file(s) in res-mdpi should be scaled up. This is exactly what happens if I create an equivalent native Android app.

The Titanium app, however, shows both of these images at their original, unscaled size.

Does anyone know why this is happening?

Thanks,
Zsombor

  • I think you might have a misconception as to what Ti.UI.SIZE does. This will show the image using the SIZE of the image. It will not scale, it simply creates an ImageView with the image 100x200.

    If you want different sizes, then you must provide the images pre-scaled to the size you want in the directory for each resolution.

    If you DON'T want to do that, then you could use percentage (ie. a percentage of the screen size) to have it scale, or you could calculate the size you want based on the density that Titanium returns.

      In this case the images are not scaled - language may have not been clear in the much earlier answers. So I will try and clear it up in a quick way.

      • Assume we are only talking about the width of a device
      • The following example variations occur in all densities
      • MDPI width would be 320 pixels 320 dp (100%)
      • HDPI width would be 480 pixels 320 dp (150%)
      • XHDPI width would be 640 pixels 320 dp (200%)
      • XHDPI width would be 768 pixels 384 dp (200%)
      • The last two are different actual width but the same densities.
      • The second xxhdpi display in this case would be 64 dp (128 pixels) wider than the first one
      • Your image will remain 200x400 for both xhdpi examples, your ui however will be able to use the extra space
      • You should not expect the device to scale the width of these two to match each other

  • My $0.02 on the whole size issue.

    Malcolm is right on – set your default units to 'dp' when you first set up a project. I tried dealing with iOS in dp and android in px. It wasn't worth the hassle, and Ti today seems to do a great job if you want to just treat everything as dp. (Question for Appcelerator devs: why isn't this just a default setting in every new Titanium project? Why should we even have to think about this for more than 30 seconds?)

    Build your graphics appropriately for the various resolutions. Write automation scripts using tools like ImageMAgick to convert high-resolution or vector graphics into 0.75x, 1.0x, 1.5x, 2.0x, and 3.0x versions for your ldpi, mdpi, hdpi, xhdpi, and xxhdpi devices (not to mention the iOS devices, which correspond now to mdpi (non-retina), xhdpi (retina), and xxhdpi (iphone 6 plus)).

    Don't rely on things like SIZE and FILL. In 4 or 5 apps with hundreds of views, I'd be surprised if I used those more than 3 times. They really don't seem very necessary to me, and now that I read your post, I'm glad I stayed away from them. Set the width and height on your views explicitly, or give them gravity to the left, right, top, and bottom. I think the use of these constants is where your example broke down. If you had set your ImageView to 100x200, your images would have behaved as expected.
