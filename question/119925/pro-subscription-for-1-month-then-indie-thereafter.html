  • Hi, I read your post about your problem. I can work with you to fix it for $199: I estimate I'd be able to do it alone in a little over two hours, which makes it worth my time. Multi-threaded programming is complex and it's possible that your code is pushing a lower api past its specification. I believe it's best to go with well worn design patterns for this kind of work.

    I'd re-write and partly re-architect the logic with you:

    • Not start/stop the timer, and not create new temp timers continuously. A design pattern for this multi-threaded scenario is to check a flag variable within the thread loop, which is never stopped.

    • I'd examine switching from a pull scheme to a push scheme.

    • I'd only allow the threaded loop to fire off a new http request once the previous one has returned.

    A great reference for this kind of multi-threaded programming is Concurrent Programming in Java: Design Principles and Patterns.
