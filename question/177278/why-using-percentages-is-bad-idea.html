    • I fully support this answer, I am am that "Malcolm".

      Percentages are bad as there is rarely an actual need for them that cannot be performed using the techniques shown in the link above. Here are two more related answers I have given;

      • images in Android
      • Images path - Android vs iOS (Alloy)

      Here are a couple of examples why using percentages as bad

      • The developer is trying to get their app to look great on their one and only Android device - so they end up with percentages like; 21.56%, 89.289% < actual percentages I have seen in this Q&A. This means the app will fail to display correctly on other aspect ratio devices - there are lots of different aspect ratios as we all know.
      • Trying to determine a percentage for standard UI elements; navigation bar, tab bar and so on. These have fixed sizes determined by the OS and the UI of that OS; Android has constantly tweaked tabs, action bars, sliders and more. iOS pre and post version 7 are entirely different in subtle ways in heights.
      • Pretending all devices are the same and your app should look the same on all is moronic and fails to treat your user or the users device with the respect it deserves for being different.

      A good example where NOT using percentages makes your app work so much better with all devices is as follows;

      • You have a windows with a table and a footer bar underneath which has buttons on it.
      • You designed the app for iOS
      • You designed the app BEFORE the iPhone 5
      • When the iPhone 5 came out the app now has 176px or 88dp values more height
      • So with percentages the footer bar height increases and so does the buttons inside
      • But WHY should they? The UI does not change, the button height specs from app did not change. But by using percentages your app did. Now it looks weird and somewhat amateurish.
      • So what should you have done?
      • The footer View at the bottom should have a fixed height in dp, it should have a bottom value of 0. The TableView should be set to Ti.UI.FILL and a bottom value equal to the height + top + bottom of the footer View.

      Example;

      • Titanium without the Alloy Framework
      var tbl = Ti.UI.createTableView({
    bottom: 50, // height + (optional top or bottom) of footer View
    height: Ti.UI.FILL,
    width: Ti.UI.FILL
});
var footer = Ti.UI.createView({
    bottom: 0, // height + (optional top or bottom) of footer View
    height: 50,
    width: Ti.UI.FILL
});
      • Titanium the Alloy Framework
        • TSS
      "#tbl" :{
    bottom: 50, // height + (optional top or bottom) of footer View
    height: Ti.UI.FILL,
    width: Ti.UI.FILL
},
"#footer": {
    bottom: 0, // height + (optional top or bottom) of footer View
    height: 50,
    width: Ti.UI.FILL
}
      • View
      <Alloy>
    <Window id="tbl">
        <TableView id="tbl">
            ...
        </TableView>
        <View id="footer">
            ...
        </View>
    </Window>
<Alloy>

      This way the bar remains the same height, the buttons inside remain the same height, but the app uses the extra space for the TableView - which is what is meant to happen. This also works EXACTLY the same for Android.

      The only thing done here is fix the height of ONE element - as that is MEANT to be fixed - everything else sorts itself out.

      This can also be extended to work with top, middle & bottom as well as left, centre, right. With middle and centre expanding to fill up the remaining space.

      There are few good reasons to use percentages as I have said, but that does not mean you shouldn't use them. If you need three columns then 33%, 34% and 33% makes perfect sense.

      But percentages should be the EXCEPTION and not the rule.
