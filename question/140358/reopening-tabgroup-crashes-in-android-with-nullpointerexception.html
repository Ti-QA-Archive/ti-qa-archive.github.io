Reopening TabGroup Crashes in Android with NullPointerException

I'm experiencing a crash in my Titanium 2.0.2 GA mobile app on Android where reopening a Tab Group is crashing with a NullPointerException.

The symptoms are very much similar to these Jira's I've found but I've tested the test code in those Jiras and it does not crash, however, my code crashes every single time I test it.

  • "Reopen" tabGroup causes "activity reference is invalid, removing from activity stack" NullPointerException
  • TIMOB-7573
  • TIMOB-7240
  • TIMOB-7572

I have confirmed that the above jiras, using the test code provided, are indeed fixed.

My application is a collection of modules (messages, contacts, maps and calendar), and each module is configured into an array. The app then creates the modules from the array and stores the returned window objects in another array which is processed when constructing the home screen of icons. Clicking on an icon on the home screen opens the window of that module.

When you click on the Contacts module - the only one with a tabgroup, the application opens the tabgroup, but if you go back to the home screen and then click on Contacts again, it crashes.

The code that sets up each module is here:

csu.module.moduleManager = {};

// Loop through the list of modules and include the 
// main modulename.js file if enabled.
var modules = csu.modules;
var config = csu.config;
for (var i = 0; i < modules.length; i++) {
    var _module = modules[i];
    if (_module.enabled) {
        Ti.include(config.moduleBasePath + _module.name + "/" + _module.name + '.js');
        csu.module.moduleManager[_module.name] = createModuleWindow(_module.name);
        Ti.API.debug('Adding module: [' + _module.name + ']');
    }
}

function createModuleWindow(moduleName) {
    var window = csu.module[moduleName].createMainWindow();
    return window;
};

  • Chris – you've done an admirable job in researching this problem and presenting information to the forum.

    Your code snippets are very lengthy and complex, with a lot of interdependencies and possibly external dependencies. It's a lot to expect a volunteer to process.

    I would suggest that you boil this problem down to a minimal case, one that fits in a single app.js with as little superfluous code as possible. In the process, you may expose the code that is causing the crash, and you may find a workaround. Even if you don't, you'll give us a simpler example that we can inspect and maybe even run in a debug session for you.

    I think if I were building an app like yours, I would avoid using a TabGroup. Android is very finicky with tabgroups; my impression is that they're good as the primary UI structure in your app, and only if that structure never changes. In other words, the TabGroup is the main navigational element, and you never change the tabs. In your case, maybe you could build some sort of view of buttons (style them to look like tabs if you want), and those buttons could change out various views. So they'd behave much like tabs, but you don't get all the baggage of the TabGroup.

    That's not to say you can't do what you're trying to do – I don't know. I've never tried using TabGroups in that way, but from reading these forums for the past 9 months, I have seen a lot of people struggle to do dynamic things with TabGroups on android.
