  • my suggestion is to attempt to isolate were in your application the bottleneck is an then attempt to optimize the code.

    you are doing alot of string manipulation and database stuff repeatedly to create each row item. you also do not have a className on you rows, which is a must.

    If it were me, I would strip the row creation method down to bare minimum, create functions for alot of the stuff you are doing and then start adding the function calls in, one by one until you determine what is causing the performance issue

  • Why are you using a unique className for each row?

    Do they have different layouts?

    Think on a className like on a template. You set this up and the os will use the same layout to render all of the rows that are sharing it.

    Try to use clear dimensions for the elements inside the row (where is possible). Each "auto" will force the device to make some calculations, this eats cpu.

    And finally if you don't manage to get the results you are looking for use this gist made by Dawson to create a table that adds more rows on scrolling. https://gist.github.com/810391
    However this will break your search so you will have to figure out how to fix it :) ( hint: cache the results from the db in a json object).
