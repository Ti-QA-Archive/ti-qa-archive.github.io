  • Hi

    Other than the obvious - test, test and more tests there are a few simple things you can do.

    This idea is based on logging those things that do not work or fail to complete;

    • Create a CommonJS module that will store and event name and the time/date
    • Call this function each time you start a process that is expected to be completed by the app itself and return the internal process id. Consider using a local database to store this information.
    • Once the process has completed remove the log entry passing back the process id.
    • Consider including a type parameter to aid your future log detection.

    Any entries still in the table must have failed to complete.

    When the app starts up check if the log table has any entries, if there are any, build up the email contents from the rows of the log table offer the choice to the user so they can email it. Obviously only add this option for testers (the app has not been deployed to the app store).

    The aim here is that you log a process that would be started and stopped in your own code block. Think press a button, so log the start, do something and then log the completion. This should not be used for things that do not start and stop in a single process - so not when you add three buttons and then later when they are clicked as there is no association between the start and stop of the processes.

    Whilst you can have the Google Android service tell you about uncaught errors - chances are the log information is so indiscernible that it will not help much.

    Google Analytics and Flurry only log getting to somewhere and then somewhere else - they will not tell you a process failed unless you parse that information afterwards.

    • Another thing to consider is app testing, check out TiMocha. It allows you to create a module that tests modules and process inside the app - you say what things should be and then it tests against it, change those values and test again.

      The idea is that you open CommonJS modules, widgets, controllers and more, then set values - confirm they are set, run the methods and functions, confirm the changes expected have occurred.

      You create these scripts and then add to them as you build the app. Re-run the tests often to make sure all is well.

    • The self created logging solution works particularly well for Titanium apps as it points out which section of your code and obviously any device features being requested caused an issue that stopped the app from functioning.

      For example if you extended the example to include manufacturer, mode, screen size and density as well as free memory and other useful things in the email then you could work out if the camera code you have fails every time on a Samsung S3 when the memory is less than 10% but everything else works - this gives you a real starting point to work with.

      With Android devices being one of the most fractured platforms around - every assistance helps out. Whilst device crash logs can be useful - they often reference things that developers have no direct knowledge of as they are system underpinnings we have no control over.
