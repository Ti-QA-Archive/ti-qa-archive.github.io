ScrollableView is automatically repositioning itself to currentPage=0 when Alloy Model updates

hi, ive tried a few different (using Alloy) approaches but not been able to get this working so wanted to ask for some help.

ive got a ScrollableView, whose Label(s) contents are updating, as the Model's data is updating (every X seconds).

The ScrollableView, has 2 Views embedded. What's happening, is that if the user has swiped & is looking at the second View (in the ScrollableView - ie currentPage = 1) and then the Model data is updated via data binding - then the ScrollableView is automatically respositioning itself back to the first View (ie. currentPage = 0).

Obviously this is very annoying for the user, who then needs to swipe back to currentPage = 1.

I've tried turning scrollingEnabled OFF and using "swipe" event to transition to the second View (currentPage=1). Not ideal as it doesnt fit with how a user expects this UI element to act. However didn't work anyway, as soon as the Model updates the ScrollView repositions straight back to currentPage=0

Is this expected behavior, or a bug ?

Also appreciate any suggestions for an alternative approach.

Using:
latest Titanium Studio, build: 3.1.1.201306112235
OSX ; 10.8.3 ; testing in iOS simulator


<Alloy>
        <TableViewRow id="LBProw" dataId="" model="{alloy_id}" >

                    <Label id="heading1" text="{heading1}" />
                    <Label id="value1"   text="{value1}"   />

                    <Label id="heading4" text="{heading4}"  />

                </View>

    I understand the concern here and I think it is a current limitation of the view model binding. I stress current because I believe this is something that can be effectively done in the future to fix it, but not in Alloy's current state. The reason this is happening is because when updates happen and the data binding re-renders the UI, it's reseting the state of the ScrollableView as well apparently.

    Would you be so kind as to create a Titanium Community Jira Project ticket with all the details you have listed here, including a link to this question. Also, be sure to give it the label "alloy". At that point I'll be able to work this into the schedule and see if I can get it fixed in the near future. In the meantime you would probably do best to not use the dataCollection binding and just manage interactions between your models/collections and your ScrollableView manually. Should not be terribly complex.
