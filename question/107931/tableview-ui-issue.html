    • This creates a new className for every row, effectively defeating the optimizations of reusing a class. The above is obviously just a part of the code since it does not include a title or other objects to actually display anything in the row. So it is hard to guess from the code provided if unique class names are really needed. But typically there are at most a few styles of rows in a given table, and using the same className for each style of row (even if the object values are different) can yield performance improvements. I would not expect it to solve the residual lines, but it might. Does each row contain the same objects? Try naming a single name for all table rows.

      — commented February 2nd 2011 by Doug Handy

    • From what I see in the screenshot there is no need for unique classNames.
      Use one name only, as Doug said to let the os do it's magic and reuse the template for the rows.
      Also you could try to set a height for the tablerow. If you know the size for your UI elements it's better to set it ( height, width ) to help the os to render the element without the need to calculate dimensions.
