TextFields in ScrollView cause buggy "jumping" in iPhone app?

I've created a ScrollView, with a View in it, with multiple TextFields (and Labels) on it. When clicking on a field partially out of the screen (usually with a keyboard up, reducing the visible area) the view scrolls to make the field properly visible, however it seems to "jump" more than it needs to sometimes before scrolling to the correct position.

I've replicated this with the KitchenSink demo, in the "textfield_scrollview.js" file, by changing the "Top" values so all 3 TextFields fit on the page (I wanted to make sure I wasn't using ScrollViews and Views incorrectly).

Here's a video of it, which will probably make more sense: http://www.screencast.com/t/gxXbKZhT3Bq (note half of the "top" TextField being momentarily visible when switching to the "middle" TextField from the "bottom" one).

Does anyone know if this is a bug in Titanium (two different "scroll" logics happening?), or the iPhone emulator? Will it do like this on the real device?

Any input would be greatly appreciated - thanks =)

    You're correct in the thought of two different scroll logics occurring at the same time. You see, you don't necessarily need to use a scroll view in order to make the contents of the entire window scroll. When the contents of the window extend beyond the bottom page border, the page will scroll automatically so long as you haven't set up the view to not scroll at all.

    As far as the scroll view goes, think of it as like a textarea. You put it on a page to make a portion of the page contents scroll in its place, not to make the entire window scroll. An example would be, you want the top half of your page's content to scroll while the bottom half stays as static text.
