V1.8 Upgrade Experiences

I have tried for a couple of days to run my application in 1.8 on Android but have given up and reverted to 1.7.5 until I have more time to investigate.
The speed of fastdev is a real plus, however, these are the things I have found inconsistent. (Admittedly, some do have relatively easy workarounds):

  • The "for each" syntax is no longer recognised

  • TableViewRows don't seem to behave as before. (I use insertRowAfter with a dictionary of custom properties to keep track of data linked to rows, but the row that gets created does not keep those custom properties.)

  • Splash screen does not show

  • the currentTab property has a null value

  • I also had to revert due to new bugs, but there are things you can do to minimize those issues. I definitely didn't have any issues with the splash screen, but TableViewRows were (and always have been) a mess for me in terms of performance and stability.

    1. Perform a clean install. Remove all the Titanium Studio directories from your computer, including the .titanium directory in your OS's user home directory. (IMPORTANT: Back up your 1.7.5 SDK directory before erasing Studio. I had to build it from source a few days ago because I made that mistake. Not hard, but time-consuming.)

    2. Make sure you know the syntax differences between V8 and Rhino. Most people won't run into them, but they're worth investigating.

    3. Create a new app and copy your Resources directory into it. This has solved a lot of problems for me in the past.

    4. Titanium has many layers. Your app < JavaScript runtime < Titanium SDK < Android/iOS < *nix. The least-stable and least-tested of these layers are, of course, your app and the Titanium SDK, although a bug could exist in any layer. You need to do experiments with clean apps to determine if your code or if Appcelerator's code is causing the problem. Just create a new app and test the functionality that's screwing up for you. In your case, you might want to create a new app and see if the splash screen works.

    — commented January 4th 2012 by Stephen Tellis

  • My experience is that sticking with old releases is best for business.
    You know the bugs and maybe, luckily, you've found some (tons of) workarounds.
    Moving to new SDKs is always stressful. Many view just refuses to open AND/OR present bizzarre behavior, and a new witch hunt season begins.

    In 20 years of development, this is the worst tool I've ever used.
    I've got stuck with 2 projects sold on this platform and MUST make it work.
    After that, so long AppCelerator, see you never again.
