Runtime error when module specified in tiapp.xml

I'm having a strange issue where I have added a module to my project by adding the following line to my tiapp.xml file:

<module version="1.0">ti.mymodule</module>

And when I build with this command titanium run –platform=iphone I get the error:

[ERROR]
[ERROR] Error: Traceback (most recent call last):
File "/Users/me/Library/Application Support/Titanium/mobilesdk/osx/1.7.3/iphone/builder.py", line 1148, in main
execute_xcode("iphonesimulator%s" % link_version,["GCC_PREPROCESSOR_DEFINITIONS=__LOG__ID__=%s DEPLOYTYPE=development TI_DEVELOPMENT=1 DEBUG=1 TI_VERSION=%s %s" % (log_id,sdk_version,debugstr)],False)
File "/Users/me/Library/Application Support/Titanium/mobilesdk/osx/1.7.3/iphone/builder.py", line 1066, in execute_xcode
output = run.run(args,False,False,o)
File "/Users/me/Library/Application Support/Titanium/mobilesdk/osx/1.7.3/iphone/run.py", line 39, in run
sys.exit(rc)
SystemExit: 65

If I remove the module line from my tiapp.xml file and run again it starts in the simulator and it says the module has been loaded. (even though it's not listed in the XML file anymore), so it seems the build was fine it was just the point when the simulator starts after build that is erroring.

If I ever delete the build folder and rebuild I have to add the above module line back into my tiapp.xml file let it build and then display error, then remove the line again before attempting to run.

It seems that the module line is needed for building the app but causes the app to break when running the app.

It's worth noting that I'm using the command line tool titanium.py to build/run the app as I'm not using Titanium Studio.

  • In my experience this relates to the module created and the Project and Target architectures by default being set to armv7 i.e. there is no armv6 branch. This is an issue with SDK's up to 1.7.2 to my knowledge. This can be overcome by opening the module project in xcode and manually adjusting the settings in both Project and Target. More interestingly upgrading to 1.8.0 creates the module project correctly and there is no need for manual adjustment of the architectures
