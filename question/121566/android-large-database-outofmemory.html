  • As it seems you know, the install method will copy the file from the resources folder to another location which is not read-only on the device. This means you actually consume twice the storage necessary, and you can't delete the resource copy after the install because the folder is read only.

    With a db of that size, I would recommend doing one of two things instead of using the Ti.Database.install method:

    • Don't include it at all, and download from a server on first use
    • Zip the database in the resources folder, then extract on device

    In either case, you need to know the path and filename of where Titanium will expect to find the "installed" database. Place the (uncompressed) file there and you can use the Ti.Database.open method instead of install().
    The method I use to determine the path and filename is here.

    Once you know that, you can either use httpClient to download the db direct to the final destination. Or if that is not practical for you, extract a compressed copy of the database from the resource folder to that location using this for android and this module for iOS.

    • I don't think the move method can work here, because that would require removing the file link from the resources folder. On devices, the resources folder (and everything below it) is read-only. So I think the move is bound to fail. Perhaps it copies 3K of data before it figures that out? Never tried it myself.

      There is no documented copy method, as you note. It seems often there are more things available then show in the official API docs, so I guess you could try a .copy() and see if one exists. But the way I have copied files in the past is to get file object references to both the source and destination files, then use something like:

      db_dest.write(db_source.read());

      Caveat emptor: I have never tried that on a 15.5MB file, and you may be back to the original error of the install() method running out of memory.

      The last time I needed to copy a file, the new filestream support of 1.7.x was not available. If you can't get by with the simple statement above, using filestreams you could now setup a loop and read say 1MB chunks at a time to read from the source and write to the destination. Or smaller than 1MB to minimize memory usage.

      However, instead of doing that I'd urge you to zip the database in the resource folder, then extract it to the desired location. This would push off the incremental file writing to the zip decompression routine.

      As mentioned in my original reply, to decompress in android you should take a look at this code. With a 15.5MB resource file, I really think it is the better approach to take here.

    • A few comments. First, on your outstream.write you don't want to specify 0 for the offset each time. I believe that will continually overwrite the first 1024 bytes of your output file with each 1K chunk as you read it. Try changing that line to simply:

      outstream.write(buffer);

      and I think Titanium will take care of the rest for you.

      Second, is the outstream "f" file object really on the SDcard? The .error text at the top of the last code block mentions the SDcard. You don't want it there, as the database should be in internal memory. Use the "f" setup from your attempt to use .move – that is, you want:

      var f = Ti.Filesystem.getFile('file:///data/data/' + Ti.App.id + 'databases/', 'newWorld');

      Third, if you want to try the httpClient again, take a look at this thread as it shows how to avoid the out of memory error in Android for large files. Although this thread was specifically about Android, for the sake of the archives and people who read this in the future, in iOS it is even simpler to download direct to a file. Set the httpClient object .file property after the .open and before the .send, as shown in the kitchen sink file download example.

    • I don't think you need to if you do what I suggest above. But while looking for that thread to reference for you, I also stumbled on a thread which said there is an undocumented variation to the Filesystem.write method in android, which lets you append to the end of a file:

      f.write(bytes,true);

      Adding the second argument and setting it to true switches the write() to append mode in android (but I don't think iOS). So you may be able to use that in your onDatastream event, without even needing the filestream. Make sure you delete any previous file before the httpClient send operation.
