Alloy - re-architect a web-centric app with Alloy; best practices, tips/tricks?

Hi all,
I have a cross-platform app in which the UI was designed by a web developer. Lots of link-style navigation sometimes 5 levels deep. Somehow, I need to salvage this app using the existing design (for now) because it was agreed to and signed off by the stakeholders before I arrived on the project AND without any input from a mobile-first developer.

I can readily convert the UI elements to alloy, but I'm encountering a roadblock for navigation. Those from whom I inherited the app used Matt Apperson's MASlidingMenu. I can choose something else if needed, however. But other than that, it's "click this link", go to a child page, "click another link", and so on. There are about 20 or 30 main views, each with conditional views. Also, tabgroups, navgroups, or action bars were NOT designed into the application. So I'm stuck with a large number of views that currently require custom navigation. That's the back story.

My question: What are some accepted best practices - or tricks of the trade that you use - to manage view navigation that also efficiently manages memory? I'm thinking along the lines of a hidden tabgroup that manages the views behind the scenes. But I don't know if that's a viable approach not having done something like that before. Or do I use windows and create controllers on button click event listeners?

This question may be too vague for a "fix my code" style forum, but I thought I'd throw it out here anyway. I'm really not sure how to proceed with this issue.
