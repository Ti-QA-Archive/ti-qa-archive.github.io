    What do you expect of a good module structure? Who says what is a correct structure?

    I don't think that there is one single correct structure for CommonJS modules.
    There is nothing wrong with your solution. Why shouldn't it be the right way?

    I personally would prefer another invocation pattern with your example. Let's look at the db object. You are defining some properties there that you don't want to export. You are working around this by assigning an object to exports.db that has only the "public" methods of db. A better way of "hiding" properties and methods of an object is using closure. Your db object would look like this (abstract):

    var db = (function(){
    // private
    var privateValue = 0;
    function privateFunction(){
         // you cannot call db.privateFunction
         // but you can call it from inner functions
    };

    // public
    return {
        getValue: function(){
            return privateValue;
        },
        someOtherFunction: function(){
            return privateFunction();
        }
    };
})();

exports.db = db;

    This is Douglas Crockford stuff. In this case db is an object with a getValue method (as an example, representing isConfigured, setAdmin etc.). These methods can still access the stuff of the outer function.
    Dig deeper into this by reading "JavaScript - The Good Parts" (I can really recommend the book).

    I also recommend you to keep a global reference in alloy.js:

    Alloy.Globals.db = require('georep').db;
