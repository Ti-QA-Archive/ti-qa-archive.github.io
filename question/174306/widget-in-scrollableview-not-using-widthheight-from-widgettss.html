Widget in ScrollableView not using width/height from widget.tss

I have the following setup:

widget.xml

<Alloy>
    <View id="view">
        <Label id="lbl"/>
        <Button id="btn_send"/>
</View>
</Alloy>

widget.tss

"#view":{
    borderWidth:1,
    borderColor:"#000",
    borderRadius:10,
    width:150,
    height:150,
    left:0,
    top:0
}

index.xml

<Alloy>
    <Window>
        <Label id="lbl_updateTime"></Label>
        <ScrollableView id="scroller">
            <View></View>
        </ScrollableView>
    </Window>
</Alloy>

when I use the following commands in my index.js
index.js

var widget = Alloy.createWidget('order', "widget", {});
widget.lbl.text = "test";
$.scroller.addView(widget.getView());

the widget size is the full height and width of my scrollableview. But if I add another view around the widget:

var widget = Alloy.createWidget('order', "widget", {});
        widget.lbl.text = "test";
        var v = Ti.UI.createView({});
        v.add(widget.getView());
        $.scroller.addView(v);

the size of the widget is the correct size from the widget.tss. Is this the correct behavior or did I do something wrong?

System:

  • Ti 3.2.3
  • Android

    Hi

    Short answer - yes.

    Longer answer;

    This is the expected behaviour. Each view added to the ScrollableView as a 'page' will have its width and height match that of the ScrollableView itself.

    It does make sense when you think about it, the ScrollableView is not a canvas but more a specialised object.

    • widget.js

      var args = arguments[0] || {};
$.lbl.text = args.title;
if (args.action) {
  $.btn_send.addEventListener('click', args.action);
}

      Usage

      $.scroller.addView(Alloy.createWidget('order', "widget", {
  title: 'test',
  action: function (e) {
    alert('hello');
  }
}).getView());

      There are a few things going on here that are different to your own code - which was perfectly fine. This gives you some other choices and things to note for the future.

      • The label is now updated outside the widget, this is good if the widget serves more than one function in this app.
      • You can repeat this logic for any other parameters you have.
      • You do NOT have to do this
      • Sometimes you want the widget to be self contained, sometimes it would be great if action inside the widget could be pushed back to whatever called it - this is called a callback. This is a simple e ample of own, the button press is now returned back to the controller that opened it.
      • You can repeat this logic for any other event listeners or other generic actions you have.
      • You do NOT have to do this
      • As we do not need access to the widget object outside of this callback friendly syntax we do not need to create a variable first, you can create all of it inline.

      Just some notes and ideas.
