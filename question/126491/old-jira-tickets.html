    • Matt, I'm talking in general. There are tickets that are very old, for example:

      http://jira.appcelerator.org/browse/TIMOB-60

      That one affects Titanium version 0.5!

      There are about 1700 opened tickets and it is hard to track and find useful tickets in that mess. I think old tickets should be checked and closed if they are not valid anymore.

      Also, some older tickets are IMHO badly prioritized. For example, this one: http://jira.appcelerator.org/browse/TIMOB-3378 is reported in version 1.6, it has "trivial" priority and fix version is "TBS" (to be scheduled). That functionality is the most basic functionality that there is.

      What's the point in having "add" functionality if "remove" doesn't work? What if "add" method would not work and "remove" would work? :D

      I really don't understand how this issues can be "irrelevant" while other issues that are edge cases are being solved very quickly.

      For example, take a look at this issue: http://jira.appcelerator.org/browse/TIMOB-5388

      It's resolved immediately, while the problem is the edge case (who starts file names with underscore anyway …).

      I haven't seen this kind of ticket solving behavior in any other product. I don't mean to be offensive, I would really like to see quality in Titanium (and there are positive steps in that direction) like the one that exists in other products.

      I hope you'll find my arguments valid. :)
