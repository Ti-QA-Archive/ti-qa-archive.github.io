Titanium Studio Deleting My Files

Hi,

I have been having an issue with Titanium Studio that I thought I should bring to the attention of the appcelerator staff. If I create a new file in the app explorer window, studio remembers this in it's undo history. When I am testing changes in my app, I often run the app then hit command-z to undo the changes and run again. Because the button that launches the simulator is in the app explorer window, the app explorer window has focus when I go back to make changes. If I don't click on the file view before pressing command-z, Studio thinks I want to undo my last action (in this case creating a file). I have deleted my most recently created file two or three times this way, and have lost several hours to recreating code. If I hadn't been using git, I would have lost hundreds of lines of code.

I just wanted to suggest that Titanium Studio implement confirmation dialogues. Irreversible actions such as deleting my source files should not happen without me intending them to.

I am using Titanium Studio, build: 1.0.1.201106171427

    So, as far as I know, here's how to replicate the issue, and a workaround if it happens. It's unfortunately a default Eclipse behavior which means that it's something we're going to have to figure out how to work around. We apologize profusely to anyone who has run into it.

    1) File > New File will create a new file. This will not respond to "Undo" (see Eclipse bug here where they fixed that)

    2) Selecting a file in Project Explorer and copying it will create a file that can disappear if you press Ctrl + Z. However, you can get the contents back again:

    2a) Go into "Project Explorer"

    2b) Right-click on the root of the project in question and select "Restore From Local History…"

    2c) Select the particular resource, and if you like, select a version. Finish out of the dialog.

    I agree that there should be some sort of confirmation, but unfortunately that may not be possible to implement easily. We're investigating all the options to see what's possible.

    In the meantime, we will add "Restore From Local History…" to the App Explorer. It's not the best solution, but it should be one step users can take immediately. Please let us know if these steps help in any way and confirm how the problem manifests itself.
