    Scott

    If you can, set an explicit height for the View; 'auto' values are costly.

    From the little code you have provided (it's not a "best" usecase, if you read the guide), it's difficult to determine what your app does and how the screen is constructed. However, if you have similar views repeated vertically, it may be possible to use a tableView instead of a scrollView. The advantage is that you can use the className property to ensure the calculations involved in drawing each "row" are performed only once.

    I'm sorry, but that's all I can suggest with the information available.

    Good luck

    • Just to muddy the waters a little bit, you asked Paul if it's possible to have two tableviews on one window and the link he provided talks about why you don't want to put two table views on one scrollview.

      I believe you can put two tableviews on one window, provided you set the height of each. But if you have other objects too, you may need the scroll capabilities of a scrollview and we are back to the answer Paul linked.

      But also note that the crux of the problem described in that ticket is use of 'auto' for the heights. In certain scenarios it may be practical to use either a static height for the tables determined at design time, or to dynamically calculate the desired height at run time before creating the tables. This in turn can lead to being able to let a scroll view calculate the content height and it may be possible to pull off. However, I suspect it would work best to not allow either embedded table to actually scroll itself. That is, the entire table should be visible. Else imagine the potential confusion for a user when trying to scroll the table contents instead of the scrollview (or vice versa) and getting the opposite effect.

      Hence it is likely only practical when you have a very finite number of table rows and those rows have a static height making it easy to calculate the total height required.

      Just being pedantic here when the question asked about a window but I think you meant to ask about a scrollview – which is what Paul answered anyway. I'll crawl back under my rock now.

      — commented February 1st 2011 by Doug Handy

    • Yes, as Doug has rightly (as always! ;)) pointed out, because a tableView is effectively an enhanced scrollView, you should never put it on a scrollView. You may be able to set the moving property of Ti.UI.TableView to false to make it static, however, which would resolve this limitation.
