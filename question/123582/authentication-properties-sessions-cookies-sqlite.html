Authentication: Properties, Sessions, Cookies, SQLite?

I have not seen any Appcelerator training on how to handle user logins. Thanks to this tutorial, I have been able to successfully setup JSON authentication from my Titanium app. So after authenticating the user, what is the best way to cache the authorization?

I am using a Rails app, so I can easily use cookies in the web browser. As far as I know, Titanium has not support sessions or cookies out of the box. In Snapost, the username and password are stored as plain text in Ti.App.Properties. I saw somewhere on the Q&A that someone recommended storing the username in a local database so that you could also store a timestamp in order for the authentication to expire. I suppose you could do the same from the remote server, but it doesn't make much sense to repeatedly check with the remote server for expiration. Couldn't I just store a timestamp as an app property in the same fashion? For that matter, is it necessary for the authentication to expire at all?

Thanks for any suggestions. I look forward to posting a Rails authentication tutorial when I finish since I haven't seen much server-side support outside of PHP.

  • Hi Micah.

    I think it depends on your needs and requirements. Sometimes you need to authenticate every time, sometimes you don't. Regarding authentication data storage, I think you can use any permanent storage (Ti.App.Properties, Database, Filesystem).

    If you need quick and simple solution, authenticate user just once and store authentication data in "Properties".

  • One method I am using to authenticate to Drupal JSON services is storing the session id instead of storing the user name and password.

    So the xhr uses the system.connect method to get a session id, stores the session id as a property.

    Then the method user.login uses the session id , plus user supplied credentials and then authenticates the user.
    From then on, all xhr requests providing the session id are have the proper privileges.
