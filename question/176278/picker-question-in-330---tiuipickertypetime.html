Picker question in 3.3.0 - Ti.UI.PICKER_TYPE_TIME

OSX 10.9.4, Ti Studio 3.3.0.20140710, SDK 3.3.0, iOS7/8

Has anyone else experienced when using Ti.UI.PICKER_TYPE_TIME in SDK3.3.0 it causes an error at compile, but using Titanium.UI.PICKER_TYPE_TIME works fine. I know the release note mention mods to Picker, but I am not sure if this the expected behavior.

[WARN] :   : ERROR: Unexpected character '#' [Resources/alloy/controllers/addcampaign.js:63,63]

Now both Titanium.UI.PICKER_TYPE_TIME and Ti.UI.PICKER_TYPE_TIME work in SDK 3.2.3

works in 3.3.0

<Picker id="pickerCampaignTime" type="Ti.UI.PICKER_TYPE_TIME" />

compile error in 3.3.0, but not 3.2.2

<Picker id="pickerCampaignTime" type="Titanium.UI.PICKER_TYPE_TIME" />

    • We have another picker-related bug in Alloy 1.4.0 that might be coming into play here. See https://jira.appcelerator.org/browse/ALOY-1091. Apparently you can use only 'picker' as the ID if you're using a date, time, or datetime style picker. We're triaging that one and will get right on a fix.

      For now, as a workaround, you can use the Alloy 1.3.1 technique of setting the picker type and values in the controller rather than in the XML.

  • ALOY-1091 (runtime failures if you use an ID other than 'picker' with a date or time type picker), and the related ALOY-1094 (crashes when you use the Ti.UI.PICKER* versions of the constants rather than Titanium.UI.PICKER* versions) have been resolved and the fixes published to the GitHub repository. You can install the development build of Alloy (instructions are listed on the repo) to get those fixes now. We will include them in the next Alloy release (date is still TBD).

    Sorry. I'm not sure how these slipped past our testing. I would invite you all to please help us catch such problems in the future by testing our beta and RC versions as we release them. We typically announce those test versions on our blog.
