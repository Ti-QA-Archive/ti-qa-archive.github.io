  • Ok - I'm thinking:

    Multiple calls to the same require statement results in one module-read from file but multiple executions. But we still don't know if Ti spawns an interpreter 10 times, and if it does, we don't know if it's killed again. I think it does, but I'm not sure.

    Changing to something absurd like i < 9999999 should hopefully result in a very long file and no real increase in memory consumption. However it could easily end with a very-long-but-not-as-long-as-the-other file and an app-crash due to memory starvation.

    Also:

    We already now that with Ti.include, there is no danger of spawning new interpreters, because included files operate in the scope they were included from, ergo it's gotta be the same interpreter instance it's loaded into. We also know that we can safely put includes in conditional statements, they are only included if programflow hits them.

    I think it's pretty safe to assume that Ti.include does not actually read the file each time - don't you?

    Include and require are both built in to Ti - so there ought not be any platform differences in behaviour.

    What do you think?
