Performance penalty of excessive Ti.App.addEventLsnr and Ti.App.fireEvent?

  • Application type: mobile - with Alloy 1.2.1
  • Titanium SDK: 3.1.2 GA
  • Platform & version: iOS 5.0+
  • Device: iOS simulator and iPhone / iPad devices supporting 5.0+ - both retina and non-retina

Hi, I have come across, while reviewing, a Titanium project which has made quite a lot of use of Titanium.App.addEventListener and Titanium.App.fireEvent etc - in some cases even where it made sense to use direct function calls or use other design patterns. I suspect that excessive usage of this event handling mechanism should be avoided because it would involve calls over kroll engine - and this could have detrimental effect on performance. Just like calling Titanium variables / native methods repeatedly (such as Ti.Platform.displayCaps.platformHeight - whose value doesn't change frequently) should be discouraged in favor of getting the value once and saving it in a Javascript variable - so as to avoid repeated calls over the kroll engine/ bridge. I also think that, if we are anyway using Alloy, we can rather use Backbone event mechanism (on and trigger, etc) - because that will at least stay only in Javascript land and not make repeated calls over the kroll bridge.

Is my hunch correct?

— asked September 22nd 2013 by Archanaa Panda

  • I use App level event listeners sparingly. There ARE times they are useful. But those times are very rare. Callbacks are more efficient and allow for synchronous control.
