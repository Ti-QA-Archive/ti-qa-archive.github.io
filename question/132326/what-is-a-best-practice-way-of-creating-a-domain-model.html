What is a best practice way of creating a domain model?

What is a solid solution to developing a domain model which stores local data in a SQL database and has on demand REST based updates.

I have a need that each domain in the application take responsibility of itself while listening to events from domains it may depend on, for example sales orders may depend on customers so would need to load orders for a customer if the customer was updated.

These updates need to be queued for efficient sending/receiving of updates in the background when required but not interrupting the user.

Is this possible or should I just implement a whole synch technique where the user must initiate a request to update all data at once? This is undesirable though.
