    The answer provided previously have been the benchmark to solving this cross platform image issues. However they were created some time ago, so do not take into account; newer better specification tablers or Alloy.

    So first things first;

    // non-Alloy
{appname}/resources/android/images/res-ldpi/
{appname}/resources/android/images/res-mdpi/
{appname}/resources/android/images/res-hdpi/
{appname}/resources/android/images/res-xhdpi/
{appname}/resources/iphone/images/
// Alloy
{appname}/app/assets/android/images/res-ldpi/
{appname}/app/assets/android/images/res-mdpi/
{appname}/app/assets/android/images/res-hdpi/
{appname}/app/assets/android/images/res-xhdpi/
{appname}/app/assets/iphone/images/

    The above is where you place your images for Android and iOS. But as you say these do not take into account bigger and smaller device only higher and lower density devices.

    You may have noticed inside the {appname}/app/assets/android/images/ folder that there are additional folder that have extra values to those above. These form part of how you build up the extra variations.

    The Android folder variations work with a combination of several factors to help determine specific factors. These include;

    • Size
      • small
      • medium
      • large
    • Aspect
      • long
      • notlong
    • Orientation
      • port
      • landscape
    • Density (shown above in detail)
      • ldpi (not really used any more)
      • mdpi
      • hdpi
      • xhdpi
      • xxhdpi
      • xxxhdpi

    Combinations of these suffixed allow Android to select the closest match to the device it is currently on. So you could have this as a variation;

    • res-long-port-medium-xxhdpi

    I imagine you can understand the rules it will be using here. The problem is that given so many permutations in those four option sets - if you chose to provide images for every combination then the app would be bloated simply by images. This is not ideal and something that iOS has managed to avoid mostly , given the small number of variations. However even iOS is likely to bump its variations this year with iPhone 6 and possibly a aspect changed iPad.

    With tablets the biggest issue is working out if the tablet itself is large or small as well as the density. You could have a 7" tablet with 480x600 screen and 7" table with a 1920x1080 screen - these are the same 'medium' size but different densities. However you could have a tablet that is 11" with a 1920x1080 screen this could be considered a 'large' device. So you need to create graphics suitable for both. However how far you go is up to you.

    Something else to take into account, the term 'tablet' is not really honoured on all devices and it may not report back the size element correctly - so combination matching can be hit and miss.

    So you really have to experiment as you see fit and try to use the least combinations as you can. Also where possible consider use of patch 9 images to reduce the overall size.
