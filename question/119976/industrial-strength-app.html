  • It depends on what you're doing. You say you want a single code-base and you mention Blackberry, so I assume you're targeting multiple platforms. Here, in my experience, are some things to keep in mind.

    1. If you're targeting iPhone/iOS, Titanium is pretty solid. Obviously you can't use the most advanced features of the platform, but for a basic, form/dialog-driven app, it's pretty good.

    2. If you're targeting anything else, Titanium is much less solid. Regardless of the version number, Titanium for Android is beta-quality, at best. There are far more unimplemented APIs, strange crashes, etc. on Android than iOS.

    3. So target iOS first. If you can't get it working on iOS, it's almost certainly not going to work on Android.

    4. The documentation is abysmal. It's incomplete, inconsistent, and sometimes just plain wrong. Almost always you'll get the advice to refer to KitchenSink instead of the documentation. So be prepared to go poking through the sample code to find the answers you want.

    5. File bug tracker tickets. Follow up. But in the end be prepared to work around bugs yourself. We've had show-stopper tickets sitting unaddressed for months, leading to redesign on our end to find another way of doing things.

    6. If you're a decent programmer and the app is of reasonable size, it might be faster to implement it separately in Obj-C on iOS and Java on Android than to spend an inordinate amount of time working around holes in Titanium. Despite being "cross-platform", you're almost certainly going to end up with a lot of platform-specific code, except in Titanium it's going to be platform-specific Javascript talking to a sometimes-wonky toolkit layer.

  • I have not tried the blackberry beta, so can't comment on whether "it is ready for prime time" except to note that it IS still in beta. In my experience with other products, even when you first go GA you can anticipate there will be issues which were not discovered during beta and features which are still lacking.

    So having a Pro or Enterprise level support plan may be very beneficial and cost effective, if you are really targeting having hundreds of thousands of users.

    In terms of try before you buy, you can now get access to the blackberry beta for only $49/month. And with GA scheduled for late June, you may only need to buy one month (or at most two) to get you out that far.

    For payment processing, you will either need the Ti+Plus modules included with the paid support plans or need to develop your own equivalents or find / license third-part modules to perform the payment processing.

    Seems like $49 would be a pretty cheap way to try both the blackberry sdk and Ti+Plus payment modules.

    IMHO. YMMV.
