What about platform related guidelines?

Hi,

am a new incomer in Titanium, but quite used to overall app developpment (all platforms).
Currently, after I was able to build some apps, and perform deployment on serveral android devices (smartphone with 2.3 and tablet with 4.0.4), I am trying to convince some people (including me) that they should use Titanium.

Most of answers of the questions I had could be found quite easilly (although It was often not on official documentation). But there's one very question whose answer I couldn't found… about platform related guidelines. Android by example, has strong recommandations (for later release) here (actionBar example in order to point out what I was expecting in KitchenSink) : http://developer.android.com/design/patterns/actionbar.html

My problem is that, using eg the KitchenSink demo, I saw no difference between a 4.0 version of my app and a 2.3 one… I thought Titanium used native UI widgets from each SDK to build the app… so was quite puzzled about this.
So, I am wondering whether this is the standard behavior (ie one unique design per OS, whatever the target is), or related to this very demo (KitchenSink)… In that last case, should I start from scratch to design a version-related app or may I use one of the provided template and … whatever I should do with it in order to reach my goal..

And in any case, is there some "best practice" or guideline that would help a developer stick to each platform's latest design / guidelines?

Thanks in advance for reading, and in advance for any suggestion / answer.

    If you want to see 4.0 widgets, add this to tiapp.xml:

    <uses-sdk android:minSdkVersion="7" android:targetSdkVersion="14"/>

    (inside the ti:app/android/manifest element)

    Without this, you'll see older style widgets even on Android 4.

    I tried this on Kitchen Sink with Android 4.1, and it does use the newer styles, but it's pretty clear that the Appcelerator folks haven't tested it with this setting, as there are a lot of text wrapping/layout problems.

  • Bear in mind that the Kitchen Sink demo is meant to show examples of functionality across all sdk versions. In your example, the action bar that was only introduced into API 11. Its' really up to you to consume new features where available and fallback when they are not. This assumes you are targetting the entire 2.-x>4.x sdk ranges.
