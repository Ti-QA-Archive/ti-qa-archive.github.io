Titanium alloy guidelines and perfomance

I have been trying to develop an iphone app in alloy framework.I have a bunch of questions about titanium development and alloy framework.

1.In my app I am using a lot of remote http calls.whether I need to make these calls in corresponding controller or model?Any particular advantage in doing it model over controller? Or is it better to abstract remote http functionality and put it in a alloy library(commonjs) file so that I can reuse functionality?Is there any sample alloy application following the guidelines.Please give me a link?

2.If I am putting a lot of code in controller whether it will affect performance or security?

3.If I am putting a lot of code in controller it will affect reusability of the code.isn't it?

4.While running generated project file in xcode I got some warnings.whether it is the default behavior or due to the bad code practices?whether it will affect deployment in app store ?

5.Performance wise alloy framework or direct titanium development which is one better?

6.A broader question Is there any way to filter data from live server to app files so that

1)I can increase the performance of the app

2)direct exposing of client app to server response.

Thanks in advance

  • Accepted Answer

    1) There's no "right" answer as to where to put your server querying logic. Ideally you would create a data structure in a model, query the data, and populate the model with that data. But you don't have to. That may be overkill and you may just as easily execute the call in the appropriate controller. It depends on your situation and usage.

    2) Whether or not you put code in the controller does not directly affect performance or security in a fundamental sense. The code that you write could effect those things, but there's nothing inherent in Alloy that would cause performance or security issues.

    3) Reusability of code again is much like my response to #2. It all depends on what type of code you are putting in there and if you plan to reuse it. If it's common code that is not specific to the current controller, you'd probably do better to put it in a separate commonjs module that can be used anywhere in your app, or even other apps.

    4) The generated code may show warnings in xcode, but no, it will not impact your ability to distribute the app on the app store. We've had lots of developers successfully submitting alloy apps to the app store for quite some time now.

    5) Alloy is building a traditional Titanium app for you. So in general, the performance should be equal. There is some functionality in Alloy that might perform better if you write it from the ground up yourself, specifically for your app (things like the sql adapter or data binding), but these are all optional and you can use the traditional titanium code for them in your controllers if you so choose. You can use as much or as little traditional titanium code in your alloy apps as you want.

    6) Filtering data you receive from your server is dependent on the server's API, not anything that Alloy could do. Alloy has libraries in it that could help you filter data once it arrives in your app (underscore's where() function comes to mind), but at that point you've already paid the performance price of the query. You'll reduce query time by improving the server's API and filtering ability. You'll reduce your UI's render time by doing the same to the data set that you receive from the query.
