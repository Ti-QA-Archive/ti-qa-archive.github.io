Android Image Sizes - What's necessary?

Currently, I have been preparing all of our assets for iOS and android based on best practices and using @2x, mdpi, xhdpi folders, etc. Everything works fine, however, I do have some lingering, beginner questions.

Take android, for example:

  • Why not just use xhdpi images across all resolutions and specify "dp" sizes to scale everything down for the lower resolutions? Are there any disadvantages to this approach?

  • When packaging for production, does the apk contain all resolution images? Does this result in a large apk size? Or does a mdpi user only receive an apk with mdpi assets somehow after the apk is sent to the play store somehow?

    Why not just use xhdpi images across all resolutions and specify "dp" sizes to scale everything down for the lower resolutions? Are there any disadvantages to this approach?

    Because you're forcing the handset to downsample all the images, resulting in slower performance and possibly blurred or mis-rendered graphics. It would be like designing a web page in which you want to show an image in a 240x320 px block but point the img tag to a 1536x2048 graphic. Inefficient, slow, wasteful. (Flip side, on newer devices supporting xxhdpi and xxxhdpi graphics, your images would be up-sampled, pixellated, and ugly.)

    When packaging for production, does the apk contain all resolution images? Does this result in a large apk size? Or does a mdpi user only receive an apk with mdpi assets somehow after the apk is sent to the play store somehow?

    All of the images are included in the apk. Yes, it makes for a larger apk. You have a few options:

    • Use R (built-in to Android) assets wherever possible (for buttons, icons, etc)
    • Use custom fonts instead of images for buttons, icons, etc.
    • Limit or avoid graphics
    • Include only low-res, small graphics in the published apk, but have the app download resolution-specific graphics on the first launch, using them thereafter.
    • Upload separate apks for each display class and work through Google Play's cumbersome process for associating each apk with the appropriate device class.
