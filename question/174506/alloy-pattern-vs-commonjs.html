    • Firstly - yes great question. One that must be on many developers minds that have not used Alloy.

      Alloy vs CommonJS

      This is easy, they are not equal in purpose. Alloy apps can and should make extensive use of CommonJS. Titanium has evolved since it first appeared.

      • Any code anywhere
      • Multiple Contexts
      • Name spaces
      • CommonJS
      • Alloy + CommonJS

      CommonJS within Alloy is now used to provide "business rules". So the code is used to provide logic, functions, calculations. It no longer provides creates or returns any UI.

      There are two things, Alloy is about separation of logic, presentation and layout. CommonJS provides the ability to create solid, smart, memory efficient elements of an app.

      CommonJS outside of Alloy is often used to create sections of an app, including logic, layout, navigation to other CommonJS functions and style. Alloy is used to separate everything, not just sections of an app. A app requiring 50 labels that look the same within a none Alloy using CommonJS (or even without CommonJS - to be fair) will have its presentation defined 50 times. Alloy will have the layout defined once.

      What is Alloy

      Alloy is the name given the MVC framework optionally used by Titanium in the creation of app. I could spend ages telling you technically what it is - but that is unimportant to the average developer.

      Alloy is a way to create apps that are much more manageable, re-suable, focused, lean and fast.

      What Alloy is NOT

      Alloy is not an alternative to Titanium, it is not a different language. You are ALWAYS creating Titanium apps using Titanium code. Whilst Alloy uses style-sheets - Alloy is NOT using HTML.

      What are the benefits of us the Titanium with the Alloy framework?

      • Separation; you no longer use styles within the creation of View, Labels, Buttons etc
      • Separation, your actual code to create the actual app is away from the structure/layout and presentation/styles
      • Separation; your layouts with; Views, Labels, Buttons, Windows and more now focus on the type of thing you need, the style and logic are elsewhere.
      • This separation might sound confusing, maybe a little annoying when you first hear about it - but it makes creating an app so much faster.
      • Re-usability, as everything is separate your code is focused on completing the task and if that could be re-utilised elsewhere.
      • Consistency; the average app might have 5 - 6 styles for labels, so why do we redefine them every single time one is added to the display - Alloy lets you say "my titles look like this", you then add a Label as say "you are a title" later on you may change what a title looks like - all your "titles" change to suit. Whilst this might not sound impressive if you come from a HTML background - this single improvement means that your apps look more professional without any effort
      • Speed of development; I have used Alloy to create both big and small apps, but each one has been MUCH MUCH faster to develop than not using Alloy.
      • Alloy has built in optimisations; your code gets optimised during the processing stage before the actual compile happens. Trust me; JavaScript allows for things that most people do not know could occur - sometimes your app logic benefits from some of those extras.
      • Widgets; these are true separations of entire elements of your app. You can even build your own re-usable ones that are used in more than one app.
      • Themes; if you need your app to look and act differently for specific compiles, you can use themes - then change one value in one file and your app now takes over the other persona during compilation.
      • Models and Collections - you will love these
      • Data abstraction

      You have to use it to see the difference

      I could sit here writing all day explaining the benefits of the Alloy framework - but you will never truly understand until you make your first test app - not a Hello World - something that tests the features you need to work with.

      A Modern App

      These days a good app has at least these things - and they are all separated;

      • Layout
        • Windows, Views, Buttons, Labels, Switches, TableViews…
      • Styles
        • Rules on how things look, app wide, controller wide or widget wide.
      • Controlling Code
        • Dynamic elements, event handling, app navigation
      • Business Logic
        • CommonJS modules
      • Internationalisation
        • Strings file - used EVEN if the app will have only one language

      Here is another recent question on the subject

      • If Alloy seems to be a panacea, why people can continue to write in classic mode?

        You have; Stephen Feather, Rick White, Aaron Saunders all agreeing - Alloy is the way to go.

      The biggest problem of trying to explain how much better using Titanium with the Alloy framework - is that you have to actually use it to really get the point. I mean more than an hour or even a day or two - then you understand. Everything you used to do outside of your actual real code that makes your app yours is faster easier, more manageable, faster on devices and re-usable.
