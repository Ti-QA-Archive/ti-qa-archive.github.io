Named function vs. App event listener

I have noticed after creating several event listeners ( Ti.App.addEventListener(&#39;myEvent&#39;) ) that I could have simply created a named function ( function myEvent() ). With the event listener, I can only pass JSON-formatted objects. In a function, I can pass many things, including Titanium objects. Is it just that event listeners exist within the Titanium.App namespace, in which case I could accomplish the same by doing the following?

(function() {
    myApp.myNamespace = {};

    myApp.myNamespace.myEvent = function() {
        // tasks here    
    };
})();

Just trying to learn when it is best to use which strategy. Thanks!

    IMHO the benefits of the events approach is that your code is not tightly coupled; there is just a listener sitting out there that any file, object can fire an event to.

    A perfect example is one i use often, when I make a change to a database table, if there are windows, buttons or any UI elements that need to be updated, I just fire an update event and the objects now know they need to do whatever they do to update themselves… I just find that to be a cleaner approach for that specific use case

    that being said if you need to pass objects that cannot be serialized then you really do not have a choice
