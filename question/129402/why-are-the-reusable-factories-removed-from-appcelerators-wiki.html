    Sorry for the slow response; I didn't see your question. The Reusable Factories guide was removed when we reorganized the wiki and also in part due to our switch to recommending CommonJS modules. What was there needs a big rewrite. It's on my radar, but as part of a larger rewrite and expansion of this Best Practices guide.

    You might want to read through CommonJS Modules in Titanium – creating component factories with CommonJS is pretty natural.

    Factories are useful when you need to create multiple instances of a "class" (yeah, JS doesn't have classes, but the concept applies here). For example, if you want all your buttons to look generally the same you can create a button factory that will accept a button label & click handler and return an object. You can easily create a CommonJS module for that. The old guide just offered a different coding style to accomplish the same thing.

    Another point we sometimes see is that people think that using factories will automatically ease memory issues. I'm not sure if that's what you're expecting. So take this as a general statement not directed specifically at your situation.

    Check out Managing memory and Finding Leaks – the key is to destroy objects when they're not needed so that JavaScript garbage collection can clean up. This can be easier in a module/factory pattern but such a pattern is not required.

    If your app is crashing, you need to figure out why. It might or might not be due to memory issues. Check out the debugging tools topics and then do some profiling, analyze the crash logs, etc. No matter your coding pattern, you need to be mindful of how many objects you're creating, their memory use, and that you need to null them out when those objects aren't needed.
