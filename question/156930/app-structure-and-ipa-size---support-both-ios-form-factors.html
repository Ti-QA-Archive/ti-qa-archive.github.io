App structure and IPA size - support both iOS form factors

Hi,

I need to create a Titanium Alloy based mobile application for both iPhone and iPad form factors. The question is what is the best way to organize my resources as well as the declarative view code. Is it advisable to declare both formFactor's views in a single XML file with qualifiers for platform and formFactor like so - 

<Alloy>
    <Tab id="miniAppTab" title="MiniApp" platform="ios" formFactor="handheld">
        <!--iphone tab view-->
    </Tab>
    <Window id="miniApp" navBarHidden="true" platform="ios" formFactor="tablet">
                <!--ipad view-->
    </Window>
</Alloy>

And go on thus for multiple such Alloy view xml files declaring platform="iOS" and formFactor="…" to qualify which snippet will be applied on runtime.

What would be the implications of this? Will Titanium repeatedly check platform and formFactor and make a call across Kroll engine which may have performance impact? Will this mean that my packaged build and IPA file will be bulkier even if it is exclusively for iphone or exclusively for iPad?

Is there a better alternative than this or best practice in this respect?

  • Application type: iOS (both iphone and ipad)
  • Titanium SDK: 3.1.2
  • Alloy - 1.2.1

    Alloy is a wrapper around the Titanium API that is used at design time to create a separation of concerns. When you build it actually writes out optimised Ti code so the form factor checking is likely to be just a test against Ti.Platform which is no faster/slower than manually performing that check in non-Alloy code. Also remember that when building for iOS the bundle generated is a Universal App so will always include the resources for both devices (unless you build two separate Apps)
