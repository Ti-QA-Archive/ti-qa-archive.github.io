    • From a conversation with Appcelerator:

      Basically, there are two major ARM architectures supported by Android today:

      "armeabi" (also known as ARM v5)

      "armeabi-v7a" (ARM v7a)

      ARM v7a is obviously the newer architecture, and supports some fancy things like floating-point hardware acceleration, and the so called "neon" instruction set. The majority of Android (and iOS) devices from the last year or two are using this architecture.

      ARM v5 is the older architecture that is still around on some 2.2 and 2.3 devices (and is also the architecture emulated by the Android emulator)

      Android will automatically extract the correct .so file based on the architecture of the phone when the APK is installed. We ship two libraries to get the best of both worlds:
      armeabi: backwards compatibility for cheaper / older devices with newer (2.2 / 2.3) versions of Android
      armeabi-v7a: optimized for the latest CPUs, delivers the best performance.

      Technically, an end user could remove the armeabi-v7a version of the shared library (libkroll-v8.so) and still be OK, but their app would be slightly slower on newer devices than it could be (don't have many metrics on this ATM though).
