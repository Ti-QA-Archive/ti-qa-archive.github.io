Images path - Android vs iOS (Alloy)

I'm starting with alloy, and I have a doubt about the path to the images, when I create a new project the path folders like this:
Android: android/images/res-hdpi/image.png…
iOS: iphone/image.png

The problem is that for call the same image from ios I can use for example this:
image: '/image.png'
but in android I need to use this
image: '/images/image.png'

I tried to move the content of android/images to android/ but it seems that the densities not work

What is the best approach? Should I create a images folder inside iphone folder and put there my images from ios and then use always a path like this:
image: '/images/image.php'

I want to use a universal path that works in both android and ios.

    Hi

    You should read the following answers I gave some time ago that explains how to manage different layouts across different devices and platforms.

    • How to achieve same layout for Android and iPhone using Titanium
    • images in Android (also iOS)

    However; as you note your project use Titanium with the Alloy Framework. These answers were written before Alloy appears. Now whilst the vast majority of the information is identical the only real difference is where the root of the app is when talking about images;

    // without Alloy Framework
{appname}/resources/
// with Alloy Framework
{appname}/app/assets/

    Your own example actually places the two files in different location - once the virtual folders are removed. This is why you have to access them differently - as you have noted. So the important part here is to set a standard baseline of where your images go - thus your question. You do have to keep in mind however that each platform deal differently with density specific images - so here goes;

    Alloy specific

    // android specific
{appname}/app/assets/android/images/res-mdpi/here.png
{appname}/app/assets/android/images/res-hdpi/here.png
{appname}/app/assets/android/images/res-xhdpi/here.png
{appname}/app/assets/android/images/res-xxhdpi/here.png
// ios specific
{appname}/app/assets/iphone/images/here.png
{appname}/app/assets/iphone/images/here@2.png
// either platform (less common)
{appname}/app/assets/images/here.png

    You then access those images as follows;

    // tss
"#something": {
    image: "/images/here.png"
}
// Ti create
var img = Ti.UI.createImage({
    image: "/images/here.png"
});

    Always use the prefix / - this is a rule you must follow.

    As you will note from the answer I linked to above there are many virtual folder. Virtual folder simply, means one that collapses when a certain set of matching rules occurs.

    When the app is on iOS the iphone virtual folder collapses so the image paths do not take that into account. On android both the android and density virtual folders collapse, with the correct density folder being used depending on the device the app is running on.

    Alloy will only include the platform images for the compiled platform, so do not worry that Android images are present on iOS or the other way around.

    Android densities and how iOS densities fall in between them can be confusing at first, so here is a quick list to explain.

    • Android
      • ldpi = 75%
      • mdpi = 100%
      • hdpi = 150%
      • xhdpi = 200%
      • xxhdpi = 300%
    • iOS
      • standard = 100%
      • retina = 200%
      • (watch this space) = unknown yet

    The logic works like this. Your images should be based on those percentages, so when creating images, base the size on 100% and then create versions and the next sizes. So if your image needs to be 200,200 then you have the following images;

    • Android
      • ldpi = 150x150
      • mdpi = 200x200
      • hdpi = 300x300
      • xhdpi = 400x400
      • xxhdpi = 600x600
    • iOS
      • standard = 200x200
      • retina = 400x400

    You will notice that the two iOS images match two of the Android images - so that saves some time. Also it is now rare to support ldpi so less need to create those. If you have suitable tools, always create any non-photographic images using a vector based program and then convert to a rastered version.

    In an ideal world you use both Adobe Illustrator and Adobe Photoshop, pasting AI objects as smart layers into Photoshop. Either way ALWAYS make your images much larger than you need today and scale down as the devices are increasing in resolution and you will need to remastered images sets as the new densities are announced.

    I also said (watch this space), read these two posts I wrote to understand why;

    • Rumour: Will iPhone 6 have a resolution change?
    • Resizable iPhone and iPad Simulators

    This should have you covered on everything you need to know on layout, images, sizes, scales, virtual folders and much more. Sorry it is so much to read and four additional links each of which are long :-)
