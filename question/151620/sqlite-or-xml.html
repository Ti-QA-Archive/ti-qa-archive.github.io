sqLite or XML

We are building a content managed app that downloads a lot of data as XML for the apps pages from a server…
Would it lead to a better performance, to read the XML and store it in a database and then query the database for the relevant data, or just read the XML directly?

  • Well, it depends on many things.

    • The complexity of your data structure
    • Will this data be searchable?

    If you have complex data and you are planning on giving your users the ability to poke the data in different ways like that. Then yes you would be more advised to store this data into a database.

    If you are just displaying the data and doing nothing later with it, then there is little point in storing the data on the device other than keeping it for offline use.

    Hope this clears this up for you.

      Traversing the file will be a performance hit every time and then the memory issue you have already considered. The indexing of the database will mean your results are instantly (relatively speaking) retrieved.

    • I am not sure the XML > JSON conversion on the server is the right plan. I am going to assume the file is a dynamic data feed, so the conversion would need to be done for each connection. Many types of XML cannot simply be exported out as JSON without server side processing after the fact and I would much prefer having 1000 clients processing data as required than 1 server processing 1000 files before sending them out. Distributed process power wins each time.

      Whichever way you look at it XML NEVER lived up to the promise - JSON is better. But sometimes you have to work with what you have got and if this is a dynamic XML file transfer to the client for processing.

      Better still bite size the data - is the entire file really needed all at the same time - if it is then is sounds more like an initial data source, if that is the case package with the app and only receive updates.
