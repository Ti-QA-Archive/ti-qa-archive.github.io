Best practice for 2 versions of same app (free/paid)

I'm doing a lite version of my paid app, but I'm not very fond of the idea of having two seperate projects with the exact same files in it. I rather check some parameter to see if some contents should be removed or not (free or paid version).
Can this be done easy in Titanium and what is the best practice for it?
I don't care if images/sounds (content that won't be used) is included in the IPA for the free version.

    Hi

    This is my understanding;

    • Lite version & Paid version as two separate apps

    Perfectly acceptable as long as the lite version is not considered a demo app, by this it means for example; if you provide 10 levels in a game those levels must be fully working levels and not have some mechanism to stop them functioning because it is a free version.

    • Single version where additional content is available via an IaP

    Perfectly acceptable as long as the lite section follows the same rules as above.

    The idea is that the user is not forced to pay to make something work - only for additional content if that is the terms of the game (or other app).

    As to your original question; "is there an easy way to manage to apps from one set of code?"

    • You need to have two different app ids, but I guess you already know that
    • The simplest method would be to add a new property to the tiapp.xml, you can then access that via the Ti.App.Properties. Use this property to determine if you are in the lite or full version.
    • I would recommend that you store all premium static content in a different folder, this way you could simply remove the folder prior to building the app or leave it there depending on which one.

    When you need to build for the store, simply edit the tiapp.xml set the appid and your own custom property, then decide to remove the static content folder.

    I would strongly consider removing the extra static content folder for the separate lite version as depending on the space it takes up it is always better for the user if they download as little as possible and there is the over-the-air limits to consider as well.

    All done.

  • This is against the App Store Guidelines. Apple wants you to have only one App in the App Store using In App Purchase to unlock content / functionality.

    So the "best practice" would be to implement IAP and set a pro flag in properties or something when a user buys or restores your pro product from iTunes.

    Actually, one single app is easier to maintain IMHO since you would not have to update two apps. Anyway, Apple will reject your Lite Version, so you better go by IAP.
