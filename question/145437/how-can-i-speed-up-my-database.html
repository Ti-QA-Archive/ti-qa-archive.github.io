  • ORDER BY is generally slow on unindexed fields. That might require a query similar to:

    • CREATE INDEX mytest_id_idx ON mytest(id);

    But from a first glance, it seems like your sample attempts to perform both DB & UI operations at the same time. Have you tried to use a CommonJS module? Separate functionalities might improve performance, specially if iterations are longer.

    Also, resultSet (var rows) should be closed just like db.close(); to enhance performance and prevent further dataBase exceptions. Simple sample:

    while (cityWeatherRS.isValidRow())
{
  var cityId = cityWeatherRS.fieldByName('city_id');
  var cityName = cityWeatherRS.fieldByName('name');
  var cityConditionId = cityWeatherRS.fieldByName('cond_id');
  var cityConditionSummary = cityWeatherRS.fieldByName('summary');
  var cityConditionIcon = cityWeatherRS.fieldByName('icon');
  Ti.API.info(cityId + ' ' + cityName + ' ' + cityConditionId + ' ' + cityConditionSummary + ' ' + cityConditionIcon);
  cityWeatherRS.next();
}
cityWeatherRS.close();//Closing my result set!

    This and some more Best practices are explained here.

    • that's a lot of data to retrieve and even more rows to render in a very complicated layout (TableViewSections). you are way beyond what a tableView can manage.
      You should think about a new way to present your data :

      -you could load your data by chunk and and let the user ask for more (use SQL LIMIT and OFFSET)

      • or if you need alphabetical order, let the user choose a letter before displaying result.

      -you could merge your first and your second row in a single custom row : you divide TableViewRow by 2 !

      • i 'm not sure to understand why you need a TableViewSection for every row : it's is used to group tableViewRows. so for example, if you want to group your data by alphabetical order, you should need only one TableViewSection by letter
