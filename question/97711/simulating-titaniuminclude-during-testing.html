Simulating Titanium.include() during testing

Hi, I'm using Jasmine as my js testing framework for mobile developement with Titanium mobile 1.5.1. It's working fine (with a little bit of hand mocking of the Titanium classes), but I wondered if there's a way I can simulate Titanium.include() to pull in other .js files into the test.

Obviously these aren't full on unit tests if I'm pulling in outside files, but there are occasions where it's expedient, nice, and the right thing to do. I'm wondering what I can use to essentially load a separate .js file similar to the way Titanium.include does it when I don't have access to the Titanium API.

Thanks!

  • I asked the Pivotal Labs folks (makers of Jasmine) what they thought and they said they had considered the common.js lib to be a good means of adding some "includey" style functions, but Jasmine strives to keep dependencies down, and so they decided against including that library.

    They mentioned if I was using the Jasmine Ruby gem (which I am), that I may have some luck by changing the order of the source files in my jasmine.yml file. That gets me the javascript ordered loading I need to make my code more modular.

    So, not quite the answer I was looking for, but it gets me there nonetheless. :) MUCH LOVE to the guys at Pivotal. Jasmine rocks, they are indeed ninjas, and really easy to talk to.

  • I had a similar issue with Ti.include() as I was putting unit tests against my code. I use the Jasmine test framework too, the version distributed with titanium-jasmine.

    Mocking out Ti.include() did not work. In fact, attempting to mock any calls to the Titanium API simply did not have any effect. I wanted to mock those calls to Ti.include() because there was procedural code executed that I needed to avoid. After restructuring my code to be completely object-oriented, this need went away. Now I have a long list of includes loaded first thing in app.js, the app starts with a procedural one-liner, and the rest of the code has fairly robust test coverage.

    I should also mention that while refactoring my code to make it testable, I implemented a simple MVC pattern. This contributes hugely to the testability of the code (including testing views without rendering them to screen) without having to mock out Titanium API calls.

    I know that this is not exactly the answer you asked for, but I wanted to share my experience.
