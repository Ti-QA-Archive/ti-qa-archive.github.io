    Hi

    The general rule is; iOS table speed is very fast, Android is usually much slower.

    Your layout is complicated - overly complicated to serve the needs of your goals. You have used Ti.UI.SIZE and Ti.UI.FILL for almost every single item - and a layout property is used on almost all parent views.

    You have even set the layout property on parent views that have only one child - thus becoming pointless.

    Much of your styles should be inside the TSS file and maximum re-use be made- again, much of what you have here is essentially unique.

    I would suggest;

    • Stripping out the styles and placing into the TSS file
    • Managing the number of variations
    • Do not rely on SIZE and FILL when you already known how much space they are going to take up.
    • Do not rely on layout properties when there is either just one child or you know where they need to be placed.

    I can assure you that with SIGNIFICANTLY more complicated row layouts, numbers of rows and child objects - performance on a 4s is excellent - even into the thousands. So it is about restraint and planning.

    BTW

    <TableView
  bubbleParent="false"
  id="mainTableView"
  layout="vertical" // totally pointless
  height="Ti.UI.FILL"
>

    But you should really being using a ListView for this process anyway.

    • The point of Alloy and all MVC frameworks is to separate the code, layout, styles and models - often more.

      By doing this your code can become much more efficient as the pre-processing can improve what you have. However you gain more advantages as YOU start to learn the benefits. For example right now you are describing the style of every single label repeatedly, however id they are all the same or mostly the same - this becomes one declaration. You can create extra classes to handle the differences.

      By using classes and naming them well you describe what your label (in this example) is meant to be showing, so at a glance you see this one is a title and the other a subtitle. With those defines in a TSS you can then tweak how ALL titles or subtitles will appear within the app or a section of the app. This means you are describing your content in the view and defining the presentation in the TSS.

      Imagine that simple point for a minute. Imagine your app had text that was all black (#000) on a white (#fff) background. But then you realised that it would be better if the white was a slightly off white (#ecf0f1) and the black a note so solid black (#2c3e50). By changing just these two values in the app.tss - your entire app will inherit those changes without you having to manually tweak each and every single usage. The same goes for padding, fonts and any other styles, being able to define an app wide style is a must for all serious developers - this allows that to occur easily.

      Because you have defined the styles within a simple class logic your app has much less code in it - so it runs faster, the pre-process can optimise if it is suitable as it needs. Within the app you can create classes that denote state; active and normal for example. Then inside alloy you can add the active class to the view/button/;abel etc which changes the way it displays and then remove it when not required. This is all in code and reacting the the app or the user. So again, one consistent pool of style definitions.

      A well designed app would always have a style guide to define what font to use for which elements, the size and weight, the colour and more. How much space between items, margins and padding - how different information can be highlighted due to importance and so on. If you are coding all your style properties in-line then any changes to the style guide by you or your client would now take a large amount of time and mistakes and omissions are likely. But using the tss files correctly - this is a matter of seconds.

      So whilst you see having it in one place is simpler - in reality it really is note and your are losing out on performance opportunities, re-usability, consistent app wide styles and readability.

      Quick example;

      "ScrollView": {
  contentHeight: Ti.UI.SIZE,
  contentWidth: Ti.UI.FILL,
  height: Ti.UI.FILL,
  layout: 'vertical',
  width: Ti.UI.FILL
}
"Label": {
  bottom: 5,
  color: "#2c3e50",
  font: {
    fontSize: 12
    fontWeight: 'normal'
  },
  height: Ti.UI.SIZE,
  left: 15,
  right: 15,
  textAlign: 'left',
  top: 5,
  width: Ti.UI.FILL
}
".title": {
  color: "#000",
  font: {
    fontSize: 14
    fontWeight: 'bold'
  },
}

      <Alloy>
  <Window>
    <ScrollView>
      <Label class="title" title="Heading" />
      <Label title="This is random content I quickly typed to allow this line of text to wrap around on to more than one line." />
      <Label title="This is random content I quickly typed to allow this line of text to wrap around on to more than one line." />
      <Label title="This is random content I quickly typed to allow this line of text to wrap around on to more than one line." />
    </ScrollView>
  </Window>
</Alloy>

      See how easy the xml is to read and how easy the meaning of the layout is? Imagine how easy it is to change the title to have a red colour and a normal weight font. Now consider the tss to be the app.tss, these styles are defined only once but used in all parts of the app you can quickly change the margins, tweak the colours and even add a custom font. But in each case only three styles have been declared here in this very simple example. Making the base label a title is done by simply adding the class "title" that adds the class entry of title to the existing label - simple.

      But do NOT think you have to change your whole app now - that is not what I want you to do or make you think this HAS to be done. On your next project yes. On the next sections of your current project - probably. Decide as you go.

      But as advice - this is the correct way, trust me - I have been doing this for a VERY long time.

      Once you have mastered this - then you can separate your static textual content into i18n files - even if you only have one language - trust me again - you will thank me.
