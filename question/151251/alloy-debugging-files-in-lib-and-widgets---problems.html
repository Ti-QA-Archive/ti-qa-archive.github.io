Alloy debugging files in /lib and /widgets - problems

I've upgraded to Titanium 3.1.0 and updated Studio to 3.1.0.
(building for iOS target SDK 6.1)
I've been using Alloy since released in December and really like the new ability to set breakpoints in the original code file rather than the derived file in Resources.

I am debugging with an iPhone simulator (6.1) on a Mac running mountain lion.

But it seems that I can no longer "step into" or set breakpoints in code located in the /lib or /widgets folders.

If I set a breakpoint in a widget folder, the debugger displays code in a random spot in my controller which called the widget. The variables in the debugger look like they belong to the widget.

If I set the breakpoint in the derived widget file in the resources folder the same thing happens.

Is this a known problem or am I doing something wrong? Any workarounds?

Thanks in advance-

    Hi Joel,
    I've tried to recreate this issue on a local project of mine. Here is what I did:

    1. Created a new widget and added it to the main view.
    2. Alloy generated "widgets/myWidget/controllers/widget.js"
    3. I've set up a breakpoint there and launched the app.
    4. The app stopped on the breakpoint as expected.

    The only think I've noticed is that the breakpoint marker on the generated file is gone when you re-launch. This is pretty much expected as the file is being touched outside the Studio and the markers that we set on it get removed because of that. However, if you'll open the Breakpoints View you will see that the breakpoint still exists and the debugger will get that breakpoint information and stop as requested.

    I would recommend cleaning the project by doing Project -> Clean, or by manually deleting the Alloy generated files under 'Resources'.
    If that doesn't help, we may need to test this using your project, so we'll need to get a copy of it (or some other test case that shows the same issue).

    Thanks!
    -Shalom G.
