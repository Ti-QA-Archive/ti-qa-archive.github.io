A Serious Issue with Vertical Layout on iOS

After finding out view.layout = 'vertical' we've used it extensively to all pages. Rather than guessing absolute positions, we laid out all elements in relative terms. So far we thought it was great.

Suddenly, a serious issue arose and it made us horrified: when you show() and hide() a view, it shows a ugly repositioning of elements. We've found that this problem occurs all the views with its vertical layout option. It's so distasteful to see how elements are getting arranged. If a view is already rendered, that's no problem. But it occurs whenever we call show() and hide(). In short, the layout feature sucks beyond one's imagination. It happens both on device and emulator.

Did anyone resolve this annoying issue? The API documentation must add a warning against layout property, if this "outrageous feature" is the default behavior of all views with that property.

  • Accepted Answer

    I have tested this in both 1.7.3 and a continuous build for 1.8.x and believe this to be a platform issue that needs resolution - track the JIRA ticket here. A watch on a JIRA ticket is seen as a "vote up", so watch the ticket as desired.

    The best path forward in this case would be to use absolute positioning for views that must be hidden/shown. Or, alternately, one could use z-indexing of view elements to first show a view which was z-indexed behind a currently visible view, and after a short period of time, change the z-index of the view to make it visible to the user. The latter is sub-optimal, but perhaps necessary if your view is dynamically created.
