  • Unfortunately I am not at home, and won't be for several days so I can't pull out the code I used specifically to delete the webview db but here is the base code I used to list the names of databases associated with my app:

    var f = Titanium.Filesystem.getFile('file:///data/data/' + Ti.App.getID() + '/databases')

if (f.exists()){
    var lst = f.getDirectoryListing();
        for (var i = 0; i < lst.length; i++){
            alert(lst[i]);
        }
    }

    This way I found the name of the database used for the webview, it was named somthing like "webviewdb" predictably enough.

    I was then able to delete by calling up the file and using the deleteFile method

    
var dbF = Titanium.Filesystem.getFile('file:///data/data/' + Ti.App.getID() + '/databases/' + webview.db);
dbF.deleteFile();

    This worked for me on android, but I have not tested it on any other platform

  • I had a think about this problem and did some investigating.

    I wondered where the cache was being stored and did a quick check to see what databases where running like this:

    var f = Titanium.Filesystem.getFile('file:///data/data/' + Ti.App.getID() + '/databases')

if (f.exists()){
    var dblist = f.getDirectoryListing();
        for (var i = 0; i < dblist.length; i++){
            Ti.API.info("db name" + dblist[i]);
        }
    }
}

    One of the names it came up with was webview.db. When I then deleted this file and reopened my webview I found that I got the shiny new un-cached version.

    This of course is the proverbial hammer to crack a nut, but it at least shows where the cached files are.

    It would be nice to have a built in function for clearing the cache though, as this can't be a rare requirement for any app that uses the webview to any extent.
