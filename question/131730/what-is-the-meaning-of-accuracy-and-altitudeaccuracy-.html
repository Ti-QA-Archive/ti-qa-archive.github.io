    Sorry for the delay on the doc updates. I had to wait for some pending location changes.

    The updated documentation is checked in for the 2.0 release, along with some significant enhancements to GPS on Android.

    To avoid confusion, there are several kinds of accuracy values:

    • The Ti.Geolocation.accuracy property is used to specify the level of accuracy you
      require. This is set to one of the Ti.Geolocation.ACCURACY_ constants. In iOS, this
      is passed more or less directly to the OS, which tries to satisfy your accuracy
      requirements. On Android, this is mapped to the distance you need to move between
      updates (a.k.a. distanceFilter on iOS) rather than accuracy per se, which leads to
      some confusion in the current implementation.

      2.0 will introduce two new accuracy constants, ACCURACY_HIGH and ACCURACY_LOW, which
      can be used for Android and Mobile Web. On Android, this gives a simple way to
      configure location updates. Under the hood, the logic is a bit more sophisticated
      and should yield better results.

      2.0 also exposes a set of manual controls for location updates on Android, for more
      demanding applications where ACCURACY_HIGH and ACCURACY_LOW don't cut it.

      If you're using the CI builds, you can play with these new modes now. Look at the
      docs for Ti.Geolocation and Ti.Geolocation.Android for details.

    • The location event includes accuracy and altitudeAccuracy properties, which as Ivan said, are in meters. Altitude is also in meters.

    • There is also a heading accuracy, if you're using the compass. On iOS and Mobile
      web, this is the maxiumum deviation, in degrees. Android doesn't give us this
      granularity; instead, the values basically correspond to constants indicating
      high, medium, or low accuracy.

    — answered March 15th 2012 by Arthur Evans
