    • You shouldn't have to do this. And I think it will cause you problems down the line if you don't keep it synced up with the generated AndroidManifest.xml (for example, if you add new activities to your app).

      If you grab those <activity> nodes from your modified AndroidManifest.xml and put them into the tiapp.xml, you should be able to make this work without a apltform/android/AndroidManifest.xml. We're doing it in our app with only the directives in the tiapp.xml

      Some of our activity nodes have this:

      android:configChanges="keyboardHidden"

      (note that orientation is removed from configChanges).

      We also have this at the very top of our app.js:

      // Force orientation into portrait so we don't show a stretched-out splash screen
Ti.UI.orientation = Ti.UI.PORTRAIT;

I wish I had documented step-by-step the exact configurations and code directives required to lock the app to portrait.     • All I can tell you is that our app is locked into portrait when we do three things:

      1. put these entries in our tiapp.xml's activity nodes
      android:screenOrientation="portrait" 
android:configChanges="keyboardHidden"
      1. put this at the top of app.js:
      Ti.UI.orientation = Ti.UI.PORTRAIT;
      1. put this in all our window code:
      _self.orientationModes = [Ti.UI.PORTRAIT, Ti.UI.UPSIDE_PORTRAIT];

      (where _self is the Window object)

      I am not 100% sure whether all of these must be present to lock the app into portrait. Next time I'm messing around with a test app, I'll test it out.

  • The issue lies with the name of the activity. Many people are using the default name which starts with a period (Ex. android:name=&#39;.MyAppActivity&#39;). You should be using the base class name for the activity org.appcelerator.titanium.TiActivity.

    <activity
    android:name="org.appcelerator.titanium.TiActivity"
    android:screenOrientation="portrait"
    android:configChanges="keyboardHidden|orientation">
</activity>
