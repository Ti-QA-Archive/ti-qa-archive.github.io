  • I think i have found the problem.

    If your app has a big size and contains many files, it wont work. So for example, if you have sound files or any other type of resources, and you have 100 (just an example) of them or more it wont work to package because their servers have some limit put in place. So after some time it just drops the connection right in the middle of the file uploading process.

    I recommend archiving, where is possible, and then un-archive when the user runs the app.

    I really think Appcelerator should look into this issue.

    I am going to install a network sniffer to see the connections being made from my mac, and i bet the packaging process will drop in the middle of the upload, because of the size of the upload.

    I will keep you guys updated.

    Alex

  • Yes, it seems that the above explanation is the problem. The bigger the size of the app, the lesser chances of it being packaged.

    I tried the following options:

    • package with 200 files - total size: 9MB - works
    • package with 750 resource files - total size: 20MB and it worked
    • package with 900 resource files - total size: 25MB - FAILED
    • package with 1500 resource files - ~50MB - FAILED
    • package with 1500 files but archived in a single file, size: 30MB - FaILED

    so this tell us clear that the packaging is limited by the size of the build, it doesn't matter how many files.

    Hope this helps.

    Alex

  • Just did a little more digging using some network sniffers and it seems that when the package is uploaded on the titanium amazon cloud the connection time depends on the cloud server is used. but if your package cant be uploaded in 2-3 minutes, i think the connection is being dropped automatically by the server.

    I kind of rounded up the allowed maximum package size to be at 20-22MB, can't say a fixed size because it takes to much time testing.
