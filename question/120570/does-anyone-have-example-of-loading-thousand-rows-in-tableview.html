    • Yes, that is what you need to do. The search bar can only filter what is in the table. It would be insane to try to put 10k rows in a table even on a desktop, let alone a mobile device. You need to roll your own search handler. I suggest a change listener, and with each keystroke use the current search value in a sql where clause, with a limit value of the number of rows you want to populate initially. Then if the user scrolls instead of typing more characters, use an offset in the sql where clause to get additional records.

    • Is the "problem" the search bar cannot filter data which is not in the table? IMHO, that is not a bug or even a design flaw. It is a "feature" because it helps make you realize that loading a large number of rows in a table in never a good thing. Design your database with the right indexes and use SQL WHERE clauses with LIMIT and OFFSET in the search field change event handler, and you can provide near instant feedback with databases with tens of thousands of rows.

  • >I don't see how the UX is improved by only loading a few rows and then loading other rows making the user wait.

    The answer to that is in loading the right rows initially, so the user never waits at all. Long before the (potential) data for a table gets close to 1000 rows, it is highly likely that what the user is interested in at any
    given time is a very small subset of the data.

    I have databases that exceed 30,000 rows. As the user types each character in a search field, a change event listener uses SQL to get the initial set of rows that match and (re)populates the table. It is literally sub-second response time, dynamically filling the table as they key each character. Generally speaking, you want to key enough letters so the entry you are looking for comes into view so you can just tap it. But I use this same dynamic scroll technique, so that if the user instead opts to start scrolling the current search results, they can scroll to find what they want instead of entering more search characters. I also let them configure their preference of how many rows to populate at once, from the number of visible rows to 50.
