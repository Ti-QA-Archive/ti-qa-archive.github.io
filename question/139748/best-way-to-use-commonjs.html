Best way to use CommonJS

I know there are a lot of threads already about CommonJS, but I haven't found anything on this specific question.

I want to create a module, which has various functions within it. Although from each, I want access to the variables in the first 'constructor' function. What would be the best way to achieve this? The titans app doesn't seem to have any functions within any of its modules and I haven't found an example that has.

Here's a couple I've tried:

This one won't have access to the self variable from within Club.prototype.Display, and I receive an error when trying to run Club.Display(); outside of the module?

function Club(){
    var ui = require('modules/ui');
    var self = new ui.View({ //view data });
}

Club.prototype.Display = function(){
    // display self
}

modules.exports = Club;

This one has access to the self variable, but is it ok to have loads of functions within the main function? it looks a bit messy. Also, how would I call Display from outside of the module?

function Club(){
    var ui = require('modules/ui');
    var self = new ui.View({ //view data });

    function Display() {
        // this works 
    }
}

modules.exports = Club;

This one I can call the Display function from outside of the module, but I cant have access to variables set within Club?

function Club(){
    var ui = require('modules/ui');
    var self = new ui.View({ //view data });
}

function Display(){
    // display self
}

modules.exports = Club;
modules.exports = Display;

      — commented July 17th 2012 by Luke Peek

    • When you're doing parasitic inheritance like that, you need to add publicFunction to _self, not this. The this variable created by the new operator is discarded if the function returns a value. In this case, the function returns _self, so _self is the new instance of MyClass. When using parasitic inheritance, there shouldn't be any references to this in your constructor.

    • Adding functions to the prototype can be a good idea, but there are plenty of situations where it does not apply.

      If you add the function to the prototype, you'll only be able to access static member variables of your class (I'm assuming that you're trying to emulate classical inheritance with your JS).

      So to attach the function once to the prototype, both _view and getView() would be defined outside the Club function:

      var _view = null;

function Club () 
{
}

Club.prototype.getView = function () {
    return _view;
}

      So _view is static – all instances of Club share the same _view. This won't work if you want to have multiple Club objects active simultaneously.

      After looking more closely at Etienne's code example, I see that it uses this, so maybe the code would look like this:

      function Club ()
{
    var _view = null;

    this.prototype.getView = function () {
        return _view;
    }
}

      This allows _view to become an instance variable, but you don't get the performance advantage from using the prototype. You're still defining the function every time you call new Club().

      In fact, you might even have some weird side effects due to scoping, where one instance of Club changes the view returned by another instance.
