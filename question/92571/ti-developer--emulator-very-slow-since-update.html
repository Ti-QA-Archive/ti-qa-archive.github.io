Ti Developer / emulator very slow since update

Since updating to Developer 1.2.2 and SDK 1.5.1 I've been having a bunch of troubles. (Windows/Android development)

My apps are extremely slow, like almost a minute to launch and 30 seconds to download a 170 byte text file from a test web server on my local network.

They've mushroomed in size. A published app of mine is 2.7 MB. I compiled it with 1.2.2 and SDK 1.5.1 and it comes out at 5.5 MB. Another app is topping out over 6 MB.

When I try to compile any of my apps using the 1.4.1 or other SDKs, I get a FAILED_OLDER_SDK error at the point when Developer tries to install the app on my emulator.

I've updated the Android SDK tools. I've restarted. I've killed various processes. I removed all my AVDs.

I'd reinstall 1.2.1 to see if that would fix the problem. But I don't have the installer…I must have deleted that file. I'm running on the same Windows XP computer I've been using for months. Since installing 1.2.2, Windows updated with a few patch tuesday updates. Maybe that's it?

Anyone else having these issues? Suggestions?

  • Titanium Developer 1.2.2 was a very minor update to the JavaScript logic for the Titanium Desktop app, and wouldn't affect your compile or deploy times.

    The binary size is much larger in 1.5 right now, as many more classes are being shipped with the SDK to support all the new Android features we've added. We're aware of this and are looking at ways to shrink our binary sizes.

    As for your issues downgrading to Mobile 1.4.x, it could be that Developer is not successfully clobbering your old project files. Removing the contents of the build/android directory (but not the directory its self) should force Titanium Mobile's build scripts to regenerate everything.

    In terms of the general slowness of the emulator, the Android tooling is generally really slow, so it's tough to say where the fault lies - and if you've updated your Android tools lately, that might be another change factor to consider. Our build process is probably also slower, since we are now looking for JSS and i18n files during the build process, and doing some additional work we didn't do before.

    If your app is significantly slower on the device (our real runtime target) then it would be prudent to start stripping out components of your app to see where the issue lies so we can attack it separately.
