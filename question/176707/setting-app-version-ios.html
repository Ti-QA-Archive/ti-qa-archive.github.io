Setting app version (ios)

Hello,

using SDK 3.2.3.GA (but I believe it's valid for any > 2.0.0) and build an iOS app, I would like to set my version number seen in app store to "1.0". From what I understand, it means that CFBundleShortVersionString must match the one define in iTunes Connect.

Now, what I found out is that the value of both CFBundleShortVersionString and CFBundleVersion are set to the value of <version/> from tiapp.xml. And, of course, this need to be a 3 digits numbers (if you give only two, the build process pads it with .0).
I have tried fiddling with Info.plist in my root dir etc but the value are overwritten in the build.

Anyone with a brilliant idea for this (admittedly unimportant) problem?

Thanks in advance.

PS. Yes, I have read the doc which says that it's a 3 digits number, but on the app store I see that apps like Twitter get away with a 2 digits number and I wondering why not me :)

    Hi

    The latest release notes for Ti SDK 3.3 might help clear this up for you;

    • CFBundleShortVersionString

    It states;

    • To generate the CFBundleShortVersionString in the application's info.plist file for iOS packages, the version field in the tiapp.xml is truncated to three dot-separated numbers. For example, '2.1' remains as '2.1', but '2.1.0.1' will be converted to '2.1.0'.
    • Prior to Release 3.3.0, the version number was always converted to three dot-separated numbers when creating a package. For example, if you specify '2.1' or '2.1.0.1', the version number will be converted to '2.1.0' for the final package.

    So for clarity;

    • You are using Ti SDK 3.2.3, as you rightly note this will take "1.0" and convert to "1.0.1"
    • When you move to Ti SDK 3.3 your "1.0" remains "1.0", the only change going forwards is if you had chosen "1.2.3.4" this will be converted back to "1.2.3"

    For information; I have older apps in the store that do have "Major.Minor", ie; "1.0" and "2.0". So the current Ti SDK 3.2.3 way must have changed from back then. But the good news is it is coming back to what you want - not what you get :-)

      • Major

        Big changes, usually massive changes - beyond a couple of features. Often when an app is rewritten.

      • Minor

        New feature, new option, including new functionality due to host OS or your API .

      • Build

        Quick fixes, bug fixes, extra little things, screen-shot changes (thanks AppStore for locking that out), compliance tweaks
