  • Hi Suresh,
    On one of my apps, I needed to implement a shopping cart and then provide options for payment at checkout (payPal or other payment processes).
    Since I was heavily into web services in my app I simply posted all the checkout information over https as a web service call to my server, where I cached the data into a temporary db table and generated a unique identifier (Actually I used a session variable) which I passed back in my web service response to the app. When I wanted to checkout I called the web page, passing the session variable as a query string parameter. On the server side I used that as the key to retrieve the data I had posted earlier using the web service, checking that the time was not more that 10 minutes ago, so that I could make sure the data was still current.

    Storing the data in the db is a requirement anyway as you will need, once you make the payment call to either payPal or the payment provider, to consider what happens if your server fails at this point. Most all payment providers have an independent back-office post mechanism that will post the response directly to your server and you can then marry up the response with the transaction in the db table. I pass the session id as a custom parameter to both payPal and other payment providers, so in the post response I can use that as the key to identify the correct record in the database.
