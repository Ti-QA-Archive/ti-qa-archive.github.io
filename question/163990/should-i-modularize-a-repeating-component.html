Should I 'Modularize' a repeating component?

If I have a repeating set of component (ie views with similar content/structure in them), should I separate that into its own controller? Then, I would use Alloy.createController(&#39;thing&#39;).getView(); to repeat it in my original controller.

I've found this to be useful for code organization, but not sure if it's good for performance? In my experience, Android crashes if you call Alloy.createController too much or too frequently (memory issue?).

Would this be a "best practice" pattern for writing controllers?

Thanks!

    • I use widgets for self contained functionality, so good examples would be;

      • Your specific way of creating a button that performs a specific function each time.
      • A Facebook style side menu
      • A common dialog library you build

      However I use them for bigger chunks of my apps. I design and define the app as specific sections of related elements. Each widget then handles those distinct sections. I might end up with 20 main sections of an app and so I have at least 20 widgets. Each widget can and often does have more than one controller each. For example a section may require a window with a table, click a row and that table calls a new window to appear with details about the record. Those are directly related so are part of the same widget. However showing a map with watching is a different section so this is a different widget.

      Two of the many good reasons for splitting up code into widgets are;

      • You clearly define the function the widget solves, provide know inputs and outputs for the widget before it is started. Your productivity goes up as you are focused on solving the needs of this single section
      • Your widget code is separate and self contained, you can stub out the inout and outputs to make a hollow version so the app can work with that version in place and when the real one is complete a - you swap it out. Helps teams work together as the shell of the entire app can be built up and then the detail gets added. Allows an early prototyping and user experience testing process to occur during the entire lifecycle.
