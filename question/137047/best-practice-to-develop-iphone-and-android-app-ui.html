Best Practice to Develop IPhone and Android App UI

We are starting the development of an Android and IPhone app and wanted to know if the best practice on developing the UI for both environments.

We have developed the core (functionality) for both the environments as common classes. The core does not have dependency on the UI at all.

Is the best practice to have separate UI code for IPhone and Android or should they be combined together and where required conditionally written?

I would like to understand this as you may have come across situations like this.

Thanks and Regards
Abishek R Srikaanth

    If you use commonJS modules for you GUI you can easily substitute components that do not work in Android.
    In a Folder structure like: 

    /Resources
-- app.js
-- MainView.js
-- SettingsView.js
-- /iphone
------/ui
--------GuestView.js
--------FriendView.js
--------/compontents
-------------FriendSelector.js
-- /android
------/ui
--------GuestView.js
--------FriendVeiw.js
---------/components
-------------FriendSelector.js

    I w MainView, and SettingView.js are used both in android and iPhone but GuestView.js, FriendView.js and the FriendSelector.js component use platform specific interfaces.

    This also helps when an iOS updated is underway, you just go to the files for the specific platform and make any adjustments.

    In General, I start with the Iphone version of everything, then step on modules that need to be replaced for android.

    Hope that helps.

  • I think it depends on the complexity of your user interface. In our app, we've been able to use a single codebase for the UI with a few conditional blocks here and there. If I found that more than 10% of the code in a particular class was platform-dependent then I might consider breaking things out more.

    On approach we've taken is to build components that isolate the main program code from any of the platform-specific stuff. For example, we have an image gallery view that's defined in a separate class. That class does have quite a bit of platform-dependent code. But the main UI just instantiates a GalleryView and uses a single set of functions to interact with it.

    To give you an idea of our application's complexity, it's about 23 class files with about 4000 lines of javascript.

  • One other thing: make sure that each time you add new UI components and/or application logic, you do some basic testing on both platforms. Don't let yourself go too far down a particular path without ensuring that you can make it work on android and iphone.

    It seems like a lot of the people who say that Titanium isn't good for cross-platform development seem to build the entire app for iOS and then start thinking about how to make it work on android. That would not be fun. Titanium is very cool, but it's got a lot of hidden gotchas. It's easier to work through them one at a time rather than an entire app's worth of issues.

    I do most of my debugging in the iphone simulator for speed, but I make sure to run the android emulator at least once a day to minimize the nasty surprises.
