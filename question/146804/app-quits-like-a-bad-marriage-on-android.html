  • This problem is now solved and here's how I did it:

    I took a cue from the warnings: "bitmap bounds cannot be determined and thus can't be scaled" thrown in the console. The fact that there were 9 such warnings led me to the culprit as the dashboard icon images. Though these were warnings but they happen to be the real deal for fixing the issue.

    First I specified the actual size of the icon image instead of using Ti.UI.SIZE to determine the size.
    Then I inserted this piece of code into the ti.app.xml:

    <android xmlns:android="http://schemas.android.com/apk/res/android">
    <manifest>
    <supports-screen android-anyDensity='false' />
    </manifest>
    </android> to take care of issue of differing display densities.

    Then I changed all the units from simple integers into integers with 'dp' units to comport with the android expectations.

    With minor tweak of the original icon size for optimal scaling, my UI now looks as expected on iPhone, and android devices (high and low density displays) in same way. Hope this is helpful to those who may have same problem (rookies like me).

    • Hello - remember me? :-)

      Obviously you have been having "fun" swap numbers to string numbers with a 'dp' suffix, there is another way.

      You can edit your tiapp.xml file to change the following setting;

      <property name="ti.ui.defaultunit" type="string">system</property>
 to
<property name="ti.ui.defaultunit" type="string">dp</property>

      This will then let you use numbers WITHOUT strings so you can, do this;

      width: 123

      …instead of this…

      width: '123dp'

      I prefer the former as the editor displays it differently from strings and I like to easily pick out the numbers. It also means the iOS and Android apps can work on the same numeric scale.

      If you want to try this, do it on a new test app and see if it helps your project or your next project or it is not for you.
