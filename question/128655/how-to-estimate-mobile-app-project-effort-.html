    Hi

    I am not that much in to time estimation stuff, its only 2 year of my experience but what i learned while estimating the app for titanium is I provide the estimation based on the number of screens.

    First i divide the screen in to 3 category.

    Normal, average and complex.

    For Normal below screen comes in it

    (Splash, Login, Forgetpassword, Homepage, Aboutus, Rssfeed, Twitter feed, youtube video feed, dashboard, custom tabmenu like wise etc)

    For Average

    If any of the above need the server API call or other advance changes.

    For complex

    (Custom Table view, Camera implementation, Audio/video recording, Third party API, Custom API call etc.., and other R&D base stuff

    I have finalized the hours for

    Normal as 4 hour

    Average as 8 hour

    Complex as 16 hour

    Now by diving the screens in to category help me count the total hours.

    This total hours can be adjusted in the later stage as some normal screen can be developed fast while other complex screen needs more time.

    But over all my estimation always comes near some time due to some unknown error take more time. But for that i always add 8 hours of time for R&D stuff.

    If client wants the app in android too then i add total screen * 2 hours for managing the layout and testing it on all devices.

    If this method of estimation you think not proper then do tell me as i want to follow the best method.

    Regards

    Nikunj

  • It really depends on the project at hand. Some screens are more complicated then others. Not all logic for all controls and how they interact is the same.

    At the end - you really have to know yourself and your capabilities and how much time you spend on every aspect of your project.

    What I do is take a project, break it to pieces, and for each piece I know how long it should take me (in hours). I also add some R&D time. Sum it all up and add 10%.

    • No, it's not true for any tool. In other tools/environments, things work (I worked with C, C++, C# and .Net, Java, PHP, JS in browser including various frameworks, PL/SQL to name some). It's not perfect in those environments, but at least trivial things work. In Titanium, they don't (at least not on Android, check the bugs I reported in JIRA if you like). It takes a lot of effort to find why something don't work because sometimes error is not reported in Studio, but can be found in DDMS as Java exception - but with no information what JS line (or even file) caused that exception. Then, sometimes events are not removed, causing "Exception on dead thread" after several window opening and closing (what's causing that can easily become mystery). Sometimes there are random bugs that are sometimes being solved with computer restart(!). There is no debugger on device, so if you plan to develop on device, debugging will be very painful. You could try to use debugger on emulator, which is extremely slow and sometime unstable (that's not Titanium problem, but it's related). It will take you 10 minutes just to start it - on overclocked quad core machine with SSD. Then, installation of Titanium and Titanium Studio - no need for explanation how painful can it be, just search through Q/A, there are tons of same topics describing problems with configurations. Then the documentation comes, which until recently was API map with basic explanations from which it is hard to know how particular part functions. Some things in documentation are wrong, some things are missing. KitchenSink is basically the best source of documentation, but sometimes it's not updated properly and doesn't work well with newest SDK versions. Also, there are no best practices decribed in documentation, not even from Appcelerator employees (well, there were/are some attempts), KitchenSink has project layout that is not recommended to use. CommonJS is implemented partially, and makes no sense to use it that way.

      So … the reasons are many as you can see. Sometimes I can do a thing in several hours if I don't encounter any bug. But sometimes, when things don't work, it takes me several days to get that work. That's magnitude grater. For any non-trivial project that takes more than a month to do, it's almost impossible to give precise timeline (it's not dependent on programming speed, but on encountered bug number, which can't be estimated). Guessing is not the game I want to play. I try to explain the situation to my manager, any reasonable man can understand the situation.

      Don't get me wrong, this doesn't mean I don't like Titanium - I love it actually. But it's far from grown up platform and it need a lot of work to reach maturity. Relying on immature platform to give (precise) deadline is frivolous for me.

      I don't know for what platform do you plan to develop, but Android side of Titanium is the dark side, for every non-trivial application you'll find a lot of problems.
