Extending the Alloy Model Functionality versus using the Controller Best Practices

I have a quick Alloy framework question for everyone. I saw in a few of the examples on GIT (in particular the login example within the Models), that there are references to placing functionality such as login/logout/authentication, etc within the model itself, rather than within the controller, and simply calling the model's functionality from the controller. I am curious if this is simply an Alloy best practice? I have come from the WPF (MVVM), and ExtJS (MVC) world, where the model literally only was a POJO like model, which contained the column definitions, but no functionality (other than basic converters and the like), and the controllers or View Models handled tying the Views to the Models, or issuing network calls. I noticed in the examples, the author mentions making network calls directly from the models, which is counter intuitive to my thinking on MVVM/MVC.

Is this an Alloy best practice? Just curious. Also, is there any performance implication here with this approach? Curious for everyone's thoughts/experience.

    Chris, good question. I think Alloy imposes the minimum in terms of restrictions to the MVC paradigm, which is why you find yourself asking this question. I would say the controller code vs. model code depends on the usage of the functionality. The "login" case is a good example it has certain functionality that needs to be available to all controllers (the logout) so it makes sense to have it in a centralized location. Whether this makes sense as a commonjs module of its own or putting it in the associated model I leave for the developer to decide. There should be no performance impact one way or the other.

    I think the models will be used more widely than you are accustomed to due to a few reasons:

    • The models themselves already carry a lot of functionality when employing sync adapters. Depending on the adapter this could be network calls, SQLite, etc…
    • Backbone.js, the core of Alloy models, lends itself to simple extension just like any other JS object. IMHO, it seemed capricious to me to arbitrarily limit that functionality. As stated before, we leave the decision up to the developer as it best fits their development.
    • Playing off what was just said, Alloy models are extremely malleable, allowing you to have them be as heavy or light as you want.

    I guess this is my long-winded way of saying that it depends on the case. But rest assured, there should be no performance concerns that depend on whether your logic is part of the controller or the model. Obviously if your model's behavior is specific to a controller, the controller makes the most sense as the home of the logic. But if the behavior spans multiple controllers, you may need to assess whether it makes more sense to include that functionality in the extension of the model itself.
