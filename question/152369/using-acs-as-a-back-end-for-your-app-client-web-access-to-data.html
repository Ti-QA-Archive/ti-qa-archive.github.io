Using ACS as a back end for your app... client web access to data.

I was just reading the new blog post here and got me wondering what others are doing. I understand the approach described in the post, but I haven't had much luck going in that direction, so have switched to creating a pure web interface for my clients.

Background: I have a couple of apps that currently use ACS as their back end and it's working great. The issue I have is I really want a good 'customer' portal for clients to maintain data stored in ACS. Most of my clients expect a 'web based' portal of some kind.

Though Titanium works great as a mobile platform, if you compile to Mobile Web it only works on WebKit based browsers and doesn't work well on desktop browsers (even the webkit based ones like Chrome and Safari). It's a tough sell to tell a client you cannot use anything BUT Chrome or Safari on my 'web interface'. Some clients only allow IE (Still!) so they need to be accommodated.

My Approach: Because of this, I have started developing a PHP/jScript based approach that is using jQuery and jqGrid to present information from ACS to a desktop web browser, allowing users to login and update ACS data.

Just wondering what others are doing? What approach you're taking and why? What tools you're using?

If someone else has already tamed this beast, it would be nice to know what the approach was.

Ray

  • Using Node.ACS, you can write a NodeJS app that runs in the ACS cloud and has full access to your ACS data. It would also have full access to the ACS APIs. Node.ACS supports the Express MVC framework.

    You might be able to find a CMS built using MVC and replace its model layer with one based on the ACS calls. That would get you a long way towards a finished solution in a fairly quick time.

    I'm going to guess that this would be pretty popular in the community. So, you'd probably be able to get help if you open sourced this and put it on GitHub.
