Deployment in an educational computer lab?

I hope this is the forum in which to ask if it is possible to deploy Titanium in a computer lab situation. Our university would like to use it in our labs and for instruction on mobile and web development. The issue is that the free version of Titanium is a single user license that requires a login on first launch. Because we use Microsoft SCCM to image our workstations, we create a template of Windows 7 and package the software to push out remotely. The result would be that the single user software would be used in the lab with 50 to 100 simultaneous logins, which I imagine would not work with the Titanium licensing. Of course students can install it on their own computers; but without the instructional opportunity in the lab, they would be less likely to adopt Titanium as their dev platform of choice.

Does anyone know if there is a way to deploy the free version in an educational computer lab environment? The sales representative was unable to answer this question. I'm looking at deployment to 300 workstations or more.

  • Each student could still create their own account.
    Simply log out at the end of each session on the system.

    In the bottom right hand corner, click the user name, and you will be prompted to log out, leaving the system ready for the next student.

    Problems may occur with managing projects.
    The answer there would be for each student to use their own drive space as the project target area for Titanium.

    • Hello again,

      I'm still chasing down an effective way to deploy Titanium in our university IT computer labs using an unattended installation. It's been a year since I last asked the question and I am hoping that there have been developments that address the issues. I recently requested for the second time that Titanium be installed in our labs. ITS investigated and came back with the following deployment and maintenance issues:

      1. Can't be silently installed. This product was not designed to be flexible in terms of unattended installations, therefore any installation we made would have to be manual and repeated whenever there is an image refresh in a lab.

      2. Installs against user profile rather than the machine itself. The problem here is twofold: software that operates like this is difficult to manage (As each machine can have many user profiles and, thus, many instances of Titanium) and the licensing situation can become rather costly.

      3. Initializing the program requires a Titanium Studio account and work is seemingly tied to this account. We would have to reset the accounts semester to semester (To prevent access to previous account holders' work) or create new accounts, neither of which is an attractive proposition.

      4. The software is dependent upon numerous subordinate pieces of software (Titanium SDK, NodeJS, Titanium CLI, and Alloy). The deployment and update of these would also be manual.

      5. Updates are stored in user accounts. Not only would ongoing maintenance of those subordinate pieces of software be manual, but we would have to perform the maintenance per each user profile on the machine due to the way that the update handler works.

      6. Given all this, I would say that the software is untenable in our environment. It appears to have been developed with a boutique environment in mind (Low number of users, limited need for automation of installation, etc.)

      Thanks for considering…
