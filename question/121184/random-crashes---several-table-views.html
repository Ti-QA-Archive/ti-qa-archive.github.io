Random Crashes - several table views

Ok, so i am a bit lost on how to debug this as it is totally random.

Titanium 1.7, iPhone SDK 4.3

I have an app with 5 tabs in a tabgroup.
All except the first have a table on them.

Two of these tabs are populated from a previously downloaded xml file.
All have child views.

The table rows have unique className.

No errors are output to the log (even in debug).
Memory seems to be ok.

Sometimes it will crash when you click the 2nd tab, sometimes you get through many clicks on different tabs before it crashes, but it crashes every time at some point.

I have a common function to read the XML files and a common function to create the table rows. There are no global variables.

The only thing common to the crashes are that i am using a custom view instead of the tabbar and calling tabGroup.setActiveTab then in the focus event i read the XML file and create the table rows before using setData to populate the tables in the tab.

Can anyone give me a clue about where to start debugging this. I could post some code but which code?

It seems to be something to do with reading the XML file for a second time.. is there an issue with closing a file handle?

Or setting the active tab.

Thoughts, suggestions,

Thanks,

Jez

  • I had issues like this where the app just crashed.

    What I found was that if you assign an event listener to the app Ti.App.addEventListener('…', action). then remove this event and add it again you get the sort of random crashes you are getting.

    Not saying this is the issue but is maybe a first place to look.

  • Ok, so i seem to have fixed it…
    The change i have made is to;-

    Download the 2 xml files from a call in my app.js and loads the info into a database on device.

    Then all pages that use the data use it from the db.

    No longer are the xml files saved atall, no longer are they being read.

    I definitely think this was down to re-reading an xml file as when i had done the main xml file and still was getting the small xml file, there was less crashes but it still happened.

    I would like to log a ticket for this as it is on the second reading of a downloaded and locally stored XML file that this fails. But it is not so easy to reproduce, so in general i will now download XML data into a local DB and call it from there.
