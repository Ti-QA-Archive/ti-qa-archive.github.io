Application design - How do you handle async requests response cases? may be "promises" related

Hello everybody,

This is not an Appcelerator Titanium specific question. The mobile app I'm building does a lot of async requests. It needs to authenticate itself, the user must be logged for some interactions, etc.

I've come to a point where any request might get any kind of response such as:

  • not authenticated
  • not logged
  • bad params
  • successful
    etc…

The thing is, I'm not familiar with this kind of app. I was wondering what are the best practices.

Some real questions for example would be:

If the app is not authenticated (token expired, first launch) should the app try to authenticate itself and then send again the request that was denied ? (transparent to user)

Should I send an authentication request each time the app launches and then "forget" about it?

The problem I'm facing is that the code becomes quickly big if I try to handle this for each request. Full of nested callbacks, retry conditions, various events listeners to manage, etc. It just does not feel very "nice".

I've been looking at the promise pattern but only know theory and don't know how to implement this in Appcelerator Titanium at the moment. So I welcome any advice regarding this particular problem. I wonder how apps like "Facebook" handle this.

Thank you for your help
