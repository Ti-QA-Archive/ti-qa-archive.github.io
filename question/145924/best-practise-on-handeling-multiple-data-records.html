Best Practise on handeling multiple data records

Hi,

i have a couple of questions regarding best practises for handling data:

I want to create an ios app for guide tours. we have around 10 tours (a tour has many routes, POI etc.) and i want to sell them via in app purchases.

  1. how to i store the data for these tours? should i have all of them in one sql DB or keep seperate ones for each tour? i want to keep in mind easiest upgradability

  2. I want to store the users purchases locally, so i do not have to query apple every time to see if they have purchased a tour. if i store this data in a sql file (e.g. preferences.db), this can be easily modified by a user ( i tried with one of my in appstore apps, and it worked with iexplorer to modify a sql file stored within an app). how can i safely keep user data on the device? i thought about using md5 hash values for properties and adding dummy data to the preference.db file to make it harder.

using cloud service is a bit over the top for this project i think, as users would have to create accounts etc.

any help would be much appreciated

  • Just a note on the in-app-purchase issue:
    Apple rejects apps that sell real-life products / services with in-app-purchase. And if you use in-app-purchase you can't avoid to query apple (or your own server) - don't store purchases locally: users may purchase products from another device.

    I would store the tours data on an application server and pull it into a local DB
