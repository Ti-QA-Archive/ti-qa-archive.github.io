ImageView issues with Android

Hello there !

I'm having some issues with ImageViews with Titanium SDK 1.7.1.

According to this post, since SDK 1.7, there is a new (pretty nice) way to manage images with Android.

So, in my code I've got things like :

var image = Ti.UI.createImageView({
    // Some properties
    image: "/images/path/to/my/image.png";
});

and it works perfectly for some images in my application. Great !

But for some other images, it doesn't work, and when I look in the FastDev Process traces, I can see:

[WARNING] [2011-07-19 12:12:34,921] get images/path/to/my/image.png: path not found

coupled with a Bitmap bounds could not be determined. If bitmap is loaded, it won&#39;t be scaled in the Android emulator console.

It seems Titanium is looking at a directory named "images" at the root of my project or (even worth), in my current directory, whereas it should automatically go into /android/images/[density-specific directory]/.

Does anyone got an idea or an answer to that issue ?

Working with:

  • Mac OS X
  • Titanium Studio
  • Titanium SDK 1.7.1
  • Google APIs Android 2.3.3

  • The 'other' images have an absolute path?
    Also try to disable the fastdev, compile once without, then re-enable it. Sometimes fastdev looses new images (added later in the project).
    You can disable it in tiapp.xml

    http://wiki.appcelerator.org/display/guides/tiapp.xml+and+timodule.xml+Reference

    search for ti.android.fastdev

  • Ok, some news before locking this thread.

    I used the way suggested in this post in order to use the old way to manage images in android.
    When thinking about it, maybe my folder structure was probably wrong: I had this:

    images
- high
-- image
-- folder
--- image
- medium
-- image
-- folder
--- image
- low
-- image
-- folder1
--- image

    and maybe the good one was:

    images
- high
-- image
- medium
-- image
- low
-- image
- folder1
-- high
--- image
-- medium
--- image
-- low
--- image

    meaning that the "high - medium - low" folders must be present in every sub-folders of my "images" folder.
    Couldn't test it though.

    I got many other issues with my images making my application crahs, due to the image files' size, but finally was able to get through.
