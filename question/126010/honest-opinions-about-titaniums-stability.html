Honest opinions about Titanium's stability.

I'm currently developing a major commercial mobile app in MacOSX Leopard, using Titanium 1.7.3, and it seems the more complex my app becomes the more I run in to crashes. I'm wondering if I could get some honest opinions from other developers about your experience with the stability of Titanium.

When I first started the app crashes were pretty infrequent – maybe once every couple of days – and often I could solve the issue by clearing the build folder. Now that my app has many more screens and features I'm running in to crashes several times a day. Sometimes I can fix the problem by clearing the build folder, but other times that doesn't make any difference. It seems that most of my crashes have to do with screens that have TableView, but most of my screens have TableViews, so it's hard to tell for sure.

Sometimes the app will crash when I try to place a label in a row, or in a view in a row, or if I decide to set a "top" attribute, and the only way to "fix" it is to figure out some other way to position the label so that it doesn't crash. Or to give a very recent example, I was just now doing some coding in a table view, and I changed the value of a constant and suddenly my app started crashing on startup. Oddly enough, when I added a "Ti.UI.Info()" statement in a for loop the crashing stopped. If I take it out, it crashes again. Very strange, and since there are no useful error messages (just the usual "Application has exited from simulator") debugging is totally trial and error, and sometimes the solutions make no sense, but they just work.

So far I haven't found any show stoppers with the crashing, but I'm starting to get a bit concerned that the application is becoming less and less stable. Has anyone else who has made a significant app with Titanium experienced this level of stability? Or has Titanium been generally stable for most devs?

  • I just happened upon this while doing a search to find out why Titanium constantly says something like this into the Android log:

    GC_CONCURRENT freed 1975KB  37% free (and then some other crap about the free memory)

    The 37% never changes, but this is almost like a leak in and of itself because it is eating the battery, though it appears to be an automatic 'garbage collector'…Stable? No, obviously.

    SO MY ANSWER IS:

    The Android part of Titanium still needs some work, or they need to more clearly document how to properly release memory, and leave it up to the developer to manage memory, providing us with a leak detection system for Android.

    iPhone stuff seems to work a whole lot better and it's easier to manage, but not due to Titanium's work on it. Rather, Apple is just better, so it's easier. Pull out Three20 for iPhone and enjoy Apple, instead of fighting with Titanium when in this case an abstraction layer is not needed.

    Having said that, if you try developing Java applications for Android in Eclipse expecting it to be easier than Titanium, you would be sadly very wrong. The TI abstraction layer really makes things easier for Android.

    And then there is the multi-platform argument, so all-in-all, not a bad solution here, and stability is mostly dependent on your ability to actually program code.

    If I had it to do all over again, I would have saved time by just doing native development, because my app is INTENSELY complex. But then, I know Objective C and Java, having already done Android and OSX native apps. So that fact puts me in a small group, and for most, Titanium is a little easier.

    The biggest misconception (and frankly this is misleading on the part of the TI group) is that you don't need to know how to use xCode/Eclipse and you don't need to know Java and/or Objective C to make apps using Titanium. You not only need to know about them, you need to be intimately familiar with them in order to really succeed. That ought to be said up front, and it's not.
