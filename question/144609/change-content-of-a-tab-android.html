Change content of a tab ANDROID

Hi,

I have a tabgroup with 3 tabs. However my app knows more than 3 pages, it knows 6 pages. 1 tab im using for settings, and in this settings page, i wish to "open another window" where certain settings are set. Well when i do "createWindow" and i open it, ill get the new window, but the whole tabgroup is hidden, this is not my meaning, because what i actually want is the following:

the selected tab is still the settings tab, BUT the information shown is not the default settings, but this other settings for which i had a new window designed.

So my question is, can i change the contents of a tabpage easily?

Ive tried lots of things, like Ti.App.settingsTab.setWindow(newWindow); (i assigned the settingsTab to Ti.App when i made it) but nothing worked.

The only other option i have left is to completely remove the whole UI ive been creating for the normal settings with a function (so a function for createDefaultSettingsUI() and a function for removeDefaultSettingsUI()). With this function the UI ive been creating will be all hidden or something, and after that i can use another function called createSpecificSettingsUI() to show the specific settings. I think this is quite an ugly way to solve this, cuz im just changing the content of 1 window where it actually should be 2 windows.

I was wondering if anyone has a solution for this, so i wont have to use the hacky timeconsuming ugly method.

Thanks in advance.

    It may seem hacky now, but compared to some of the things you will have to do to finish your app, it will seem downright elegant later! (I'm only halfway joking – you will likely have to hack your way around 3 or 4 bugs and/or undocumented feature non-parity issues to build any reasonably complex app).

    The approach you describe is the right one. A window will obscure the tabs, so you can't use a window. Instead, build each "window" in a separate view. Hide/show the views as appropriate.

    If the views are not too complex, you can build them all at creation time. If they have lots of subviews or have things like tableviews with hundreds of rows, you should probably consider building them only as needed by the user.

    • Am I missing something? Hylke seem like he wants to click a tab to display his settings window, then choose a particular setting and display a 'child' window and continue to display the active tab. I do this exact scenario without switching out the main view…sorry Jason, switching view content is not 'terrible' advice. In my case the UI and code for the Child settings window can be complex. Combining many settings windows into one giant settings UI with a lot of code could quickly become unruly. Everyone should be using a CommonJS style approach to writing code.

      //this code runs on my main settings window (winSetttingsMain)

      //the active tab is Settings tab at this point

      var _winSettingState = require('/ui/settings/setting_state')).SettingsStateWindow();

      _winSettingState.title = 'My State';

      winSettingsMain.containingTab.open(_winSettingState,{animated:true});

      Now my child settings window ( _winSettingState ) is displayed, and the active tab is STILL the Settings tab.

    • Two things:

      1) Using these multiple views as "windows" does not preclude using CommonJS. In fact, I think you'd find that I am one of the biggest proponents of CommonJS design – not a week goes by that I'm not "scolding" somebody for not using CommonJS. :-)

      You could put each subview in its own module. That would be extremely clean. Then all the main window has to do is manage which of these views is visible. Not unruly at all.

      2) You're missing a subtle issue, which Hylke mentioned in his initial post:

      when i do "createWindow" and i open it, ill get the new window, but the whole tabgroup is hidden

      Maybe you're using iOS exclusively and you're not familiar with the behavior of the tabgroup on android. On android, if you open a new window from a tabbed window, the tabs are covered up by the new window. This is what hylke was trying to avoid, which of course, led me to the subview suggestion.
