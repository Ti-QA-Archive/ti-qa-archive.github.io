    • Hi

      The easiest way to achieve this is to use device independent pixels (dp), this convert the numerical data into a constant layout matrix that helps to even the playing field when making sure that you can support multiple devices each having; different physical pixel widths and densities.

      As iOS and Android both use different defaults to translate numerical values into to positions on the device you are seeing those differences played out in your app.

      A short description of the differences - a fuller version can be found here Layouts, Positioning, and the View Hierarchy.

      • iOS use dp
      • Android uses pixels

      The easiest way to think of the difference is the way iOS works as standard using DP. Think iPhone, the iPhone has a pixel width of 320 pixels and the iPhone retina has a pixel width of 640 pixels. However when you are adding something to the far right of the screen you provide 320 as the answer for BOTH iPhone and iPhone retina, because the DP solution has 1 DP = 2 pixels on the retina devices. This way we only have to think about laying out our app once for iOS and both Retina and Non-retina look the same.

      This technique aid in the simplicity of providing an interface without having to worry about the details. Right now you may be thinking, "but that wastes pixels on the retina iPhone". You would be right AND wrong. You would be right as you are not referencing each specific pixels but instead a 2x2 grid on tiny pixels, but now both screen resolutions are positioned correctly. You CAN still access the actual pixels simply write 320px instead of 320 the first one is actually HALF way across the screen. Now you can access every pixel. But before you think that is great, this is not good for layouts as you non-retina devices will guess how to position that point as it would not appear in an actual location it could map.

      Android uses pixels, so here the opposite is true, you are specifying the specific pixel each time, but now you would have to manage the translation yourself for each potential scale factor of devices.

      So the easiest thing to do is to try and force them both to behave the same. But which way? If we force pixels then we have to manage a lot of scale calculations on the fly. So they easiest (but not always perfect) way is to go the dp route. This way when you specify a point on the screen it is translated for you and you simply think in terms of a virtual screen.

      If you think in terms of all devices having a virtual width of 320 wide when held in portrait then the dp solution means that you specify 0 for left, 160 for centre and 320 for far right - for ALL phone format devices. For other devices there are some small differences but much easier to consider using this technique.

      Now, whilst this is a simply solution - this is NOT the answer for all apps of all types on all devices. I am offering advice to provide the simplest and fastest solution for most phone device apps. Once you have nailed this thought process (easy to do in a tiny amount of time) you are then able to comprehend what to do in the other cases.

      So - how to tell Titanium that you want to use dp as a unit of measure. There are two main ways;

      • State your choice on each dimension, position and size;
      var txt = Ti.UI.createTextField({
    borderStyle: Ti.UI.INPUT_BORDERSTYLE_ROUNDED,
    color: '#000',
    font: {
        fontSize: '14dp'
    },
    height: '40dp',
    top: '10dp',
    value: 'Malcolm',
    width: '300dp'
});
      • However I prefer NOT having to convert all my numbers into strings and suffix them. Mostly because I am likely to forget one and that is a pain to track down, but also your code editor colour codes numbers differently from strings which makes it easier to see numbers from a distance.
      var txt = Ti.UI.createTextField({
    borderStyle: Ti.UI.INPUT_BORDERSTYLE_ROUNDED,
    color: '#000',
    font: {
        fontSize: 14
    },
    height: 40,
    top: 10,
    value: 'Malcolm',
    width: 300
});

      But to still use simple numbers you have to tell your app you want to use dp all the time as your preferred unit of measurement - good news; Titanium thought of that.

      Edit your tiapp.xml file which you will find here;

      /{appname}/tiapp.xml // this file
/{appname}/resources/
/{appname}/resources/app.js

      If you are editing this file in Titanium studio, swap to the tiapp.xml tab at the bottom of the file editor, if you are editing it in another editor you are already there.

      Find the entry;

      <property name="ti.ui.defaultunit" type="string">system</property>

      And change it to the following (if the entry did not appear look hard add the following)

      <property name="ti.ui.defaultunit" type="string">dp</property>

      Note the system value has changed to dp.

      Re-compile the app and now you can use the simple number system.

      This has ended up being much longer than I expected, if you have reached this far WITHOUT falling asleep - give your self a little clap - you did well ;-)

    • Short answer no.

      You should not expect a phone and a tablet to have the same exact layout or aspect ratio.

      All the suggestions and recommendation above are to allow you to manage the different densities of each device - not to make each device look the same.

      The logic here is that an iPhone and an iPhone retina will end up with the same layout as they are the same aspect and the same width and height. But the iPad is a different ratio and larger actual device. When designing for a larger devices you should have different layouts.

      Pick up an iPhone and an iPad using the same app and see how developers make use of the extra screen real estate.

      The (almost) only time you should ever hope that one layout fits all (ignoring the obvious aspect ratio issues) is if you are creating a game.
