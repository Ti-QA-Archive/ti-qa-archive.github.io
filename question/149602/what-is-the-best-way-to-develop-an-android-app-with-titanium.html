  • Hi

    Your question is too general for any real one answer fits all solution.

    But to deal with your specifics;

    • Testing Device

    When developing for Android - you cannot just test on the Emulator or one or two devices, you need lots, each providing different screen resolutions, aspect ratios, features, custom skins, camera resolutions, internal and sd card storage sizes. At a minimum you should have 5 devices, covering the main screen sizes. Developing for the high end only will mean your app with either perform badly on lesser devices or simply not at all.

    • Android 4.2.2

    Bad choice to target (UNLESS you have an extremely good reason)_

    This OS is not currently support by Titanium, first problem. Second problem; so few devices actually have it as the OS, your market is massively limited in size.

    The vast majority of devices run 2.3.x. Whilst 4.x is catching up, 4.2.2 or even 4.2.x is a small percentage of the 4.x branch of currently installed devices.

    Unlike iOS, Android OS upgrades are much slower in uptake for two reasons; carriers hate the idea and slow the roll-out. Most users (of older devices) are either unwilling, unable because the do not know how, or unable because the hardware is outdated and will not upgrade.

    If you are not assuming your app will run on devices with displays like; 240x320, 320x480, 480x600, 480x800, 640x800, 800x1200 to mention a few then you are in for an unwelcome surprise. Developing for Android needs planning from the start on how things will look on all screen sizes. You then need to work out if your storage requirements will exceed the average 2.x device.

    The information Trevor Ward gave you pointed you towards more information, this is a learning exercise and one that cannot be answered in a single question I am afraid.

    As to your comment to Sarafaraz about Ti SDK, always use the latest Titanium SDK, but target the Android audience rather than the latest version.

    • As I mentioned you should not target a single Android SDK, your app should aim to work on many of them. For example if you create an app in Titanium and ALL it does is show an alert then it will work on all devices. The Android SDK version should be somewhat irrelevant as if you target the more recent Android SDKs you are then disallowing older OS versions the ability to run your app. If you use older SDKs then you are allowing more OS versions but limiting some of the features.

      This is a trade off game, not helped by the fragmentation of the Android system. You CAN create one code base and compile for earlier devices using an older SDK minimum target and then with the same code base target higher devices. This will create two APK files which you are now able to supply for ONE app. So you can target high devices with one code base and provide all the higher resolution images in the file. But also create another version using the same code that does not include the higher resolution images so earlier devices can have a smaller footprint.

      This is something you will get a feel for when you start coding and working through problems.

      Everyone has their own preference on what to do and that is dictated by the expected users, anticipated hardware they will use, the type of app you are creating, what time frame you have to build, expected revenue stream from the app.

      Our basic Android testing involves these handsets;

      • HTC Wildfire
      • HTC Desire S
      • Galaxy Nexus 4
      • Galaxy Nexus 7

      These four devices cover the main device resolution testing and performance you should be reviewing as you go. You do not need THESE devices, but something like each of these to get to each screen size for real world testing - including the capabilities, particularly the HTC Wildfire as this has a tiny screen resolution, very little local storage, very slow, but a VERY COMMON handset.
