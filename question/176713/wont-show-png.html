    Now, this is interesting. Does this mean you have an image located at {project_folder}&#x2F;Resources&#x2F;Resources?

    Because that seems kind of foolish.

    If this is an alloy project, thats just stupid to put an image in the Resources folder where it might get overwrittern (you know, since your stuff is supposed to be in the app/assets folder).

    If this is a classic project, and all your stuff is IN the Resources folder, why on this beautiful green planet would anyone go and put ANOTHER Resources folder inside of the FIRST Resources folder? Geez.

    See? Foolish.

    Oh, MAYBE, you thought you should reference things including the Resources at the beginning, instead of understanding that things should be referenced from the root (which is Resources in a classic project).

    But then, you are wondering, wait, this WORKED on iOS it should magically work on Android! I did something WRONG and it actually worked, so it should work in a broken state across the board! This is pixie dust kind of thinking.

    • No, if you have a VALID (we will come back to this one in a bit) PNG located in {project_folder}&#x2F;Resources of a classic project, then it would be referenced as &#x2F;my.png and not as '/Resources/my.png`

    • Surprise! Android and iOS use 2 different javascript engines, so, they are NOT the same and do not always behave the same.

    • iOS being all unicornish will sometimes tolerate malformed pngs and jpgs named as pngs, while Android will not.
