  • Hi

    Firstly apologies to Dan and Nick as I felt this should be a separate answer - I support both of your positions but felt my own comments need their own answer.

    I am disappointed by your suggestion that Appcelerator to be useless - it seems strange that your belief that an entire development solution can be rendered useless by something that does not exist and nobody else appears to be in a desperate for.

    Whilst I do understand there is a section of the developer community that feel more comfortable using a visual editor when developing you rarely find a seasoned developer that ever uses that editor - normally they use their preferred text editor like; Sublime or Notepad++ depending on OS.

    In reality the visual side of the app is such a small percentage of the final app code that it really adds nothing to the development process in any meaningful way (more on this later). With a visual editor you can become very lazy as the hope is that the editor itself writes your code and you end up never learning how to code the important parts that links things together.

    There are thousands upon thousands upon thousands of apps being developed by more than 540,000 developers (at the time of writing) in Titanium (and the Appcelerator enterprise version), some using Alloy (framework) and some not. But almost none of these are using a visual IDE (there have been a few visual editors but non have taken off as far as I am aware).

    Coding is an art, something you create and manipulate to manage your app - your app develops as you go and grows with each need to change things. You do not create a table by dragging in on to the screen - you add data to the screen in a meaningful way that just happens to use a table. This is how you design - by working out the user experience - the user journey, how and where you gain the data to display, how and where you ask for more.

    You plan your app by sketching out potential screens, potentially using a wire-framing tool or service, you determine if what you think you need to use to show create your interface is what you should be using. You define your app look and feel and ideally maintain that going through the process of building the app, so define colours, fonts, sizes and style - all that are reused often and well considered.

    By saying Appcelerator (Titanium) is useless is without a visual IDE is insulting to all those who have produced there own apps for; themselves, their friends, their company, their clients. Some who have no budget but spend their free time creating their app, to those with enterprise level budgets with teams of developers, designers, user experience and testers.

    You mentioned above that Dan is an expert (#7) he is and a good one, you also have an answer from Nick who is also an expert (#6) and just as good - and now this one from me, I am an expert (#1) and I think I do good work - certainly my clients from small businesses to enterprise clients to military contracts and more agree. You also have (at the time of writing) a large number of negative votes on the question by those who clearly disagree with your logic.

    If you are new to Appcelerator Titanium then you should read the guides in the documentation, have a play with the examples - start with alloy - it is a really fast way to plan and develop your app and in many ways easier than an visual IDE without losing the same things I mentioned above.

    So I would like to end this with - if you are unable to use Appcelerator Titanium without and IDE then I am sorry but you will not be able to use Titanium - you will be missed as we like everyone to give the solution a good and fair go. It is a shame as you are going to be missing out on something great.

    Malcolm Hollingsworth - someone who should know what he is talking about.

  • I think that Tamer wasn't trying to offend Titanium but it's more a transliteration mistake.

    Here are a few tips to help you write professional code and successful projects:

    1- Ask for complete sketches before starting
    To make sure the client knows what he wants exactly. Ask for complete sketches and a document containing the scope and all features listed. This will create a mutual understanding of the tasks and save you a lot of time later determining if any task is inside or outside the scope.

    2- Use Alloy
    Although some developers can write high quality classic Javascript I definitely advise using Alloy. This way anybody reading the code or continuing development won't have a hard time to get up and running.

    3- Keep styling to the end
    start creating the skeleton xml and basic layout tss. You may use Jade to create the views faster http://naltatis.github.io/jade-syntax-docs/, and make it clear you're keeping styling to the end to avoid distractions.

    4- Determine when you need a widget and when a controller is enough
    You'll need widgets for example for login or any view that you plan to reuse in multiple projects or versions of the same app. Break down the screens to views and subviews and create separate controllers for independent ones.

    5- Ensure your code remains cross platform
    If you're writing a cross platform app make sure you handle appropriately platform specific code that may break the app on the other platforms. For the views use platform specific tags or folders http://docs.appcelerator.com/titanium/3.0/#!/guide/Alloy_Concepts-section-34636240_AlloyConcepts-Platform-SpecificResources, and for the controllers use Alloy directives like OS_IOS to write code for a specific platform.
    Even if you're writing for one or two platforms keep in mind that support for other platforms is on the way and make sure your code takes this into consideration. This will save you a lot of effort later, to support Windows phones for example.

    6- Organize your Javascript files
    Organize your code appropriately, for example don't put data access or API access code directly inside the controllers. Use commonJS modules for that, and arrange them to allow easy switching between different backend vendors and other data sources. This way if the client decided he wants to switch from his own managed backend to ACS the transition will be smooth.

    7- Read Titanium logs
    When running the app on simulator, Titanium provides a wealth of feedback and advices in its logs. Make sure to read the output of the console at the same time you're testing the app and address any warnings.

    8- Keep an eye on community tools
    Many active members of the community are sharing tools and tips to make other developers' life easier. Try evaluating tools like TiShadow https://github.com/dbankier/TiShadow, Ti-mocha https://github.com/tonylukasavage/ti-mocha and RapidDev https://github.com/appersonlabs/RapidDev

    9- Follow developers' blogs and twitter accounts
    Keep updated with the latest information, bookmark any link you find useful, read and understand code shared by the professionals instead of just copy pasting it.

    I hope you find these tips useful
