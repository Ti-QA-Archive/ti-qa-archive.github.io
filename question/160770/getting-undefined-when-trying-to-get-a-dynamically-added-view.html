    Hello Crystaleez,

    Problem Description

    Here we have two questions:

    • How to access accueilscreenxxx
    • How the memory management works in Alloy

    Let's go to the first question

    Accessing elements

    The way you can get access to accueilscreenxxx is either:

    • Iterate the children of $.basewincontent, until the id is the "xxx" value.
    • Add a custom variable to $.basewincontent like: $.basewincontent.customaccueilscreenxxx = accueilscreenxxx and you will access to that variable thru: $.basewincontent.customaccueilscreenxxx. In this case, you will need to null $.basewincontent.customaccueilscreenxxx and if you have other ref to accueilscreenxxx, you need to null it as well.

    Memory Management in Ti

    This is a very interesting topic, let's analyze the pattern in the Classic Titanium first.

    Memory Management Classic Ti

    1. Create element
    2. add stuff to the element (children, events listeners) show it / do something with it
    3. close it / remove it from views
    4. remove all children
    5. remove events
    6. null it

    Extra comment: Remember to remove ALL the references to any variable.

    Keep track of events

    You will need to keep track of events attached to any view/element, so you can actually remove the event listeners properly. Remember: you will need to use named functions when adding events to any view/element. You can have an accessory function like this one:

        registerEventListener(entrada, {event: 'click', callback: print_alert});
    registerEventListener(salida, {event: 'click', callback: print_alert});
    registerEventListener(falta, {event: 'click', callback: print_alert});

    So, when you add or delete any object, you call the register or unregister function:

    rh.addEventListener('click',function(){
    win2 = win2 || Ti.UI.createWindow({
        backgroundColor:'white',
    });
    registerEventListener(win2, {event: 'androidback', callback: win2Close});
    Ti.App.addEventListener('win2:close',win2Clean);
    rh_build();    
    win2.open();
});

    Register and Unregister functions defined by:

    /// Recursive Clean of Memory 
// by Mauro Parra
// https://gist.github.com/mauropm/2655813
// Begin recursive clean of memory
function do_clean(e,c){
  clean(c);
  e.remove(c);
    Ti.API.info( 'Deleted child at do_clean' );
    return;
}

function clean(e){
    if (e!=null){
        if(e.children){
            Ti.API.info( 'Number of children: ' + e.children.length );
            for(var i = 0; i<e.children.length;i++){
                do_clean(e, e.children[0]);
            }
        } else {     
            return; 
        }
    }
}

// End recursive clean of memory

/// <<< Register & UnRegister Event Listeners
// Original from: https://gist.github.com/minhnc/2333095
// Thanks minhnc! 


/**
 * params: {event: 'event', callback: eventCallback}
 */
function registerEventListener(obj, params) {
    if ( typeof obj._eventListeners == 'undefined' ) {
        obj._eventListeners = [];    
    }

    obj.addEventListener(params.event, params.callback);

    var eventListeners = obj._eventListeners;
    eventListeners.push(params);
    obj._eventListeners = eventListeners;

    Ti.API.info( JSON.stringify(obj._eventListeners) );
}
function unRegisterAllEventListeners(obj) {
    if ( typeof obj._eventListeners == 'undefined' || obj._eventListeners.length == 0 ) {
        return;    
    }

    for(var i = 0, len = obj._eventListeners.length; i < len; i++) {
        var e = obj._eventListeners[i];
        obj.removeEventListener(e.event, e.callback);
    }

    obj._eventListeners = [];
}

    Then, when you want to remove a view or element, you first remove the events and then clean everything:

// by Mauro Parra
// https://gist.github.com/mauropm/2655813
// Begin recursive clean of memory
function do_clean(e,c){
  clean(c);
  e.remove(c);
    Ti.API.info( 'Deleted child at do_clean' );
    return;
}

function clean(e){
    if (e!=null){
        if(e.children){
            Ti.API.info( 'Number of children: ' + e.children.length );
            for(var i = 0; i<e.children.length;i++){
                do_clean(e, e.children[0]);
            }
        } else {     
            return; 
        }
    }
}

// End recursive clean of memory

    Remember: Starting 3.1, you don't need to go thru all the children, since you have the function removeAllChildrren. With that function in mind, the pattern should be:

    1. Given a child a
    2. will remove the child a from the current view will remove the children of a
    3. will null a
    4. no need to do the recursively trick anymore

    Memory Management with Alloy

    1. Alloy will manage a lot of stuff for you
    2. Alloy includes some memory management
    3. you don’t need to keep a reference to secondary windows in your index controller, since…
    $.openDetail = function () { 
var detailController = Alloy.createController("detail");
// ... etc
detailController.window.open();
};
    • you will directly create the other controller, then just open it
    • when you close it, it will be removed from memory automagically, since there is no other reference to it

    In case you need a reference:

    $.openDetail = function(){

$.detailController = Alloy.createController("detail");
// .. etc
$.detailController.window.open();
$.detailController.window.addEventListener("close",function(){$.detailController = null });
};
    • if you still want a reference in your index controller, just add an event listener for close and null the child controller
      -when you close it, it will be removed from memory automagically, since there is no other reference

    In general, in your controller you will need to add a shutdown function

    • if you remember, in c++ there was a function to clean the object instance on destroy
    • you can do the same for your controllers, specifying in fine grain what you need to remove and in which order
    • you apply a removeAllChildren in the window and then null it

    When using Collections in Alloy

    • You get a really nice function, called detroy() which will free all the binding resources associated with the controller and its UI components. This is specially important when using mode/collections in order to avoid memory leaks. You call $.destroy() when you close the master window of the controller or when you want to remove it from the app.
    • WARNING: it won’t clean any UI, you have to remove it as we mentioned.

    Garbage Collector differences

    • iOS will try to remove actively any unused memory, as soon as possible
    • Android with v8 will wait until there is no more memory, and that could lead you to a “no more memory, i will kill the app”
