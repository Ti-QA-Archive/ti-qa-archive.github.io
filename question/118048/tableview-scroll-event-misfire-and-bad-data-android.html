TableView Scroll Event Misfire and Bad Data? (Android)

Summary: In essence, the scroll event misfires and does not consistently display data. When scrolling down the firstVisibleItem index is offset by 1.

Scenario 1 - Normal Table

The scroll event on TableView's doesn't fire unless scrolled over 50% of the way past a row. Once more than 50% of the row has been covered the event will fire. Is this a misfire?

Scenario 2 - Table with Set Row Heights

Now, if the height of each row is explicitly set, the event will fire once the last few (2 or 3) pixels of a row is still visible, as well as once the first few (2 or 3) pixels of the row is visible. This is fine.

However, here is the issue: there is a difference when scrolling down and scrolling up. See these two images:scrolling down and scrolling up. When scrolling down the scroll event fires at the beginning of Row 1 with firstItemVisible value of 0, at the end of Row 1–value of 0, begin of Row 2–value 0, end of Row 2–value 1, begin of Row 3–value 1, and so on. When scrolling up the values are shifted up, so when it fires at the end of Row 3 the index is 2, beginning of Row 3 the index is 2, end of Row 2–index is 1, and so on.

Scenario 3 - Fast Scrolling

When scrolling fast, the scroll event fires at random. In order to get it to fire twice per row you have to scroll slow. Can the random firing be prevented?

Conclusion

Is this a Titanium bug? Is this the way it is supposed to work? How do I consistently find out what the first visible row is? Is there a way to fire an event for every single row that comes into view or disappears?
