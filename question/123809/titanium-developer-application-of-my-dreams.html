  • Not sure what you mean by out of 'Mac experience' (or why that is particularly important) but… this will quickly turn into Titanium Developer or something similar. There are certain things you will just have to do like setting up provisioning and distribution profiles, select Titanium / Target OS SDK versions, configuring your tiapp.xml file etc. Then when you get further you will decide you need to see the console output from your app or debug etc. This is a big can of worms to recreate existing tooling with a slightly flasher UI.

    So I guess I'm curious as to what limitations Studio and Developer have that would draw you to creating an even more limited way of interacting with Titanium?

    (and if you really are keen, then Titanium Developer is available from github, easiest way would be to grab that and hack on it rather than start from scratch).

    • User experience is everything but it is not only based on UI but functionality as well, a beautiful platform conforming UI is nothing if it doesn't have the needed functionality.

      FWIW I am on OS X and use TextMate for most of my Titanium coding, Studio is really only used as a debugger and for making small changes as I come across them.

      I'm not trying to discourage you (well maybe a little bit;) but I think a much nicer approach here would be to build plugins against your favorite IDE, not very much is needed for doing a lot of day to day things where you do not need a debugger; build, run, deploy, publish macros calling through to the titanium python build scripts would get you most of the way there with some fanciness to hook console output. For things you need to do rarely (modify tiapp.xml for example or setup profiles) or the complex stuff like debugging I'd just leave that to Studio / Developer.
      And if you want to add some of that then building a standalone application that can be launched by a macro from your favorite IDE would be another approach, thinking just a simple tool that lets you modify profiles and app settings that takes a command line param to the project and another to show the appropriate dialog. Keeps it lean and mean and bindable to a key into any IDE that lets you launch an external executable.

      I would just be wary of a can of worms here, things that start off simple like running a few scripts rarely stay that way and as it gains the features to become a potential replacement it could turn into a large project in its own right.
