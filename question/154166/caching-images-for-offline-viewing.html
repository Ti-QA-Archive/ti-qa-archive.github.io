Caching images for offline viewing

Hello gurus,

I could use some general advise.

I'm gleefully developing an app that has a lot of news/media-realted lists of content. A first content feed is loading great and I'm able to generate a table of article listings (seen above). Cool!

The end goal is to cache content in a database for offline viewing (found great instructions to make that happen). I'm a bit stuck on images however…

Currently table row images are loading remotely, with each row showing up one at a time when the image loads per row. Once deleting the view and coming back to it, the remote image loading process starts again.

I'm thinking I should load all the images after the feed loads to external storage (or internal if not available), if the local images don't exist - then load internally. That would make the loading time a bit longer, but viewing time shorter correct? If doing so, I'm guessing I would need to reference in a database so that when content gets deleted, the associated images get cleaned up as well…

Anyone else run into this challenge? What do you recommend?

  • Hi

    Image caching and in fact any caching should be well thought out and relative to the use case and level of connectivity of the device. So thinking about it is a great start.

    As you have identified it might be a good idea to cache the images locally. But before you go down this road you should consider the age of the images. By this I mean how often will the user look at this image.

    • Once - If they are only only going to look once - then no caching required.
    • A few times - if these are spread apart in time then no caching required.
    • Lots of times - caching required.

    How many images are we talking, you may sacrifice speed with total size required.

    Management - you are in charge and should employ either a FIFO (first-in-first-out) clean up or a least-used-first-out process. This will depend on factors only you can decide - but keep file size as a consideration rather than numbers as you may find some images are noticeably larger than others.

    Consider adding a date stamp to the db rows to show data entry added and date last accessed - useful if you have lots to manage things.

    This is a good starting point for an image caching function.

    • cachedImageView

    Remember "Lazy people do nothing - smart people may still end up doing nothing", the difference is determining if caching is the best solution for your app. I would start by putting some logging in tot he app to determine repeating use of specific images, if this ends up being a number that requires triggering effort - then do something. If it does not, then focus any effort somewhere else that can improve the app.

    Lastly remember the OS of the device does handle image caching as best as it can, iOS is better than Android at this but both do it to varying degrees - you may not need to override this.

    Good luck.
