  • TSS is not CSS, though they share some features.

    • Separation of concerns - both let you apply stylistic settings independent of the declaration of components.
    • Cascade/prioritization - TSS lets you assign properties at the component, class, ID, and inline level, and applies priority. TSS adds the notion of an app.tss file for global properties.

    CSS features not implemented/available in TSS:

    • Complex accessors, such as child selectors (something like table>tr>td>div might work in CSS but not TSS)
    • Much of the richness of runtime/dynamic CSS is missing or limited in TSS. While you can use Dynamic Styles to add/remove a class, it's not quite as flexible as CSS's mechanisms.
    • Loading and importing style sheets (there's no @import or way to include multiple <style ..> tags in TSS)
    • Media / device queries (though TSS offers special qualifiers that accomplish a similar purpose)
    • The HTML layout engine - there is no box model, margins, flow, absolute/relative positioning, etc. in native UI layout therefore there cannot be something like that in TSS

    Let me emphasize that TSS is simply a means to apply the properties and values you see listed in the API docs. Titanium and the native platforms themselves do not use the same sort of layout engine as HTML, therefore TSS cannot. You should be comparing TSS/Titanium layout to native application UI layout, rather than HTML/CSS.

    The properties you assign with TSS are converted to "Classic Titanium" code by the Alloy preprocessor. In fact, it's important to keep in mind that Alloy is a preprocessor. Your XML and TSS is not present at runtime, having been converted to JavaScript by the Alloy compiler. This is why dynamic class manipulations are less rich in TSS than in CSS. The TSS is gone, replaced by JS objects, defining the various properties, which are applied to your UI component objects. The overhead of creating and maintaining those JS objects, plus the functions to manipulate them adds overhead to your application. These functions do not offer the richness of CSS's runtime style manipulation features.

    You'll notice that large portions of our guides and API docs show the "Classic" coding techniques rather than Alloy. This is on purpose. Alloy code is converted to Classic by the Alloy compiler, then it is that classic code that is compiled to your native application. There are guides on UI and layout where we cover the basic concepts; those apply to Alloy-based development as well as Classic. You should explore those for further information.

    On top of that, Titanium is an abstraction of the native platforms (iOS, Android, Windows, etc.). Therefore, you should familiarize yourself with the native platforms' layout and UI concepts. Things like "DP" (display-independent pixels) come directly from those native environments. We provide some info in our docs, but it's important to check out Apple's and Google's as well.
