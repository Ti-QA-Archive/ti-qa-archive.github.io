ICU Console Output

Hi,

This may be a stupid question but I have just come across something that worries me slightly.

I have an app that is being tested and distributed ad-hoc at the moment and I have been loading it on the phones using the iPhone Configuration Utility.

Someone reported an error so I used the ICU console output to see what was happening.

I noticed that as well as finding the exception that caused the error ALL the info and debug outputs in my code were also being printed in the console.

My question is:

Is there anyway of stopping this ? I would have thought that the info/debug statements would have been removed when running the Deploy to Store build option, but it appears not. This seems to me to present a security risk as it may show data, passwords, keys etc. that need to be seen during testing and debugging, but should not be available on a Deployed app (ad-hoc, enterprise or otherwise).

Or am I just missing a config option in the Deploy scenario that I have missed? - I don't really want to search my code and remove all the info and debug statements just for a deploy-able build.

Any insight on this or a point in the right direction would be welcome, even if it is a RTFM :-)

    • The debugging tools like this that output to your console are tied to things like NSLog on the native side… They are designed to not only help with debugging when building your app but to also output debug info on released apps… All apps native and not output some things to the log, but yes you do need to be careful what you output.

      As for a future release not outputting to the console when released, that will never happen as m any debs rely on this to debug released apps too.

      It has it's reasons to exist as it does, very good reasons in fact.

      There will always be APIs that when used incorrectly can get you into trouble. (see for example the "path" app for ios and the way they used the contacts API on iOS)

      Could it be more clearly documented, sure… :) maybe file a JIRA ticket for that?
