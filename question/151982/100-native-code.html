  • Hi

    The problem here is the wording of the question as it automatically brings with it ambiguity.

    • Titanium create 100% native apps
    • It uses JavaScript as the common language to allow multiple platforms including; Android, iOS, BlackBerry, Tizen and more coming.
    • The JavaScript is intelligently interpreted as the app is running and uses Just In Time dependence loading of objects.

    Strictly speaking it does not create "100% native code", to do that it would need to compile the JavaScript and then convert it into ObjectiveC for iOS, Java for Android and so on.

    However the app itself is a 100% native app using the native objects, views and hardware features.

    As to your second question "Will Titanium use our HTML5 / JavaScript source and create 100% native code for android and iOS" - short answer is no. Longer answer below.

    Titanium is a huge SDK with access to the vast majority of native everything. It does use JavaScript as mentioned - but it does not translate code you have created for another platform (HTML5 in this case) and make that "work". You will have to re-write sections of your app, including much of the UI as although there is support for native webViews these are used to show HTML content and not run apps from directly.

    You will also find problem when submitting apps to the iOS app store (Android will accept anything). iOS has rules on what will be allowed in, and if the app is basically (or could have been) a HTML5 web app, it is unlikely to be accepted into the store.

    If you are truly looking to take some pre-rolled HTML and JavaScript then PhoneGap is the better platform, you will not have to make as many changes but you will not end up with a native app (actually a hybrid web app)as the PhoneGap solutions is basically a web view that shows html content and has a very simple shell that makes it appear as an app. Great coders can turn these into great apps, but most are simply not great apps and spend a lot of time trying to pretend to be a native app.

    Remember that FaceBook recently abandoned "hybrid web apps" as the performance was very poor and they simply could not provide the experience they needed.

    • Facebook Speeds Up Android App By Ditching HTML5 And Rebuilding It Natively Just Like The iOS Version

    If you want native apps, then a solution like Appcelerator/Titanium is the way to go, but you will need to put in effort as there is no simple import HTML5 and output native app. And there shouldn't be as they are built for two different sets of platforms and goals if they are done correctly.

    • To further explain as your question like many before it assumes there is a simple answer.

      Appcelerator/Titanium creates native apps, the language it uses to achieve this is not the same one used as the mobile platform themselves.

      When writing desktop applications for Windows, should you use;

      • C
      • C++
      • Pascal
      • Cobol
      • C#
      • VB
      • F#
      • Java
      • many many others?

      Microsoft themselves have used each of the above to create apps and the underpinnings of Windows itself, yet none of these are native code - they all compile down to code that runs as required.

      Also the following; are interpreted languages from that same list; C#, VB, F# & Java - all of these are compiled to pcode - these require a run-time interpretor to actually run. Yet these are still native code development SDKs.

      This is where Titanium is - nice and squarely - it creates its own pcode which is interpreted on the fly in an efficient way so there is no real difference.

      But you then have others like Flash and Silverlight, these are specific platforms that use there own ways of doing things and do not have any required relationship with the host operating system.

      Finally you have webapps, this is code and ui elements from HTML and CSS that performs functions - but you must have a web browser that must sit on the operating system. These are drawings on a web page to mimic an app. These have their uses - and great coders can do great things - but the vast majority of people will get a web page in a browser with a workable UI.

      It is true that originally JavaScript was designed for use with browsers - but that was years ago and ignores everything is is now used on that has nothing to do with browsers.       But again NOT ALWAYS - on any of this, it is correct that all things blur and all things have their benefits and negatives.

      If you are moving a HTML5 app - I suspect Titanium is not going to be the easiest of routes.
