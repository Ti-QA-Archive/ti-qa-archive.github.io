Detect undeclared variables in .js code in Titanium Studio

Hello,

Is there any way to force in Titanium Studio that all variables are declared with the "var" clause? Or, at least, that a warning alert us about we have created a global variable without explicity using the reserved "var" clause.

The goal is prevent mistakes creating undesired global variables. I'm wondering if it could be done installing a plugin in Titanium Studio or setting a preference option that I unknow…..

For example, in Visual Basic you can use the code "Option Explicit", and I heard sometime ago that you can use a plugin in Eclipse to do something similar with Eclipse and .js files

What I meant with "global" is global in the current context. I currently use in my code the CommonJS especification, and that avoid global JS variables by itself. But using undeclared variables could get to unexpedted errors in run-time.

This is an example I want to avoid:

inline code
var myWindow = Titanium.UI.createWindow();
var currentWindow;

currentwindow = myWindow;
//After that code, we have a global variable "currentwindow" and a local variable "currenWindow" with an undefined value.
inline code

If that could be detected (or warned), we can review the code and realice that we have misspeled "currenwindow" variable.

Regards.

    Hi

    Short answer - no.

    Titanium is a set of APIs/extensions to JavaScript - so we follow the rules of JS.

    JS has huge advantages by not requiring variables to be declared and variables themselves to have no defined types. But the advantages bring the same problems you have mentioned.

    But where you see negatives - I simply see the side effects of the advantages. Plus it forces people to test their code often and make code modular as well as reducing as much as possible to small manageable chunks.

    With Visual Basic, the option explicit is a hang over to the past, it was never required to start with (in fact never existed if you go far enough back) but in time it was added as an option and later became an expectation. However these two languages are chalk and cheese, VB uses strongly typed variables (even var is a type of sorts) so option explicit enforces that rule. As the rule does not exist for JS enforcement would mean that all the advantages would be lost.

    You are already using CommonJS, so you know that small efficient code is the way to go, aim to make everything modular, define the scope of the methods and properties you are going to create before you start and then check them when you finish. Test the code in isolation and then in the actual app. Test the app often. These all sound like obvious things, but I assure you most people ignore them - get carried away - try to "save time" or think their prototype is something that is now the real app.

    I do not know how extensive your knowledge of JS is, or the Ti SDK, if you are using Alloy yet then also underscore and backbone, but there is much more to Ti/JS than a few APIs and you will come to love and understand the flexibility of the language is its greatest strength.
