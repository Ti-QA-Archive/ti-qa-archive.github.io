Blocking on Http Results - Best Practice?

At the very beginning of my app I need to make an Http call and based on the result decide if the app will continue or basically give an error message and close.

The standard httpClient approach seems to be to use asynch calls. So my question is what is the general best practice on blocking? Do I put the logic (or a call to the logic) in the event listener? Or do I put some type of loop waiting for an indicator (this seems like a bad idea). I'm pretty sure the best idea is to chain everything through the event listeners but I'm curious if others have found better/different ways to handle this.

  • Accepted Answer

    In my application that requires an httpClient call for authentication, I just display a "Progress UI Element" and do not move on until the onload handler OR the onerror handler from the httpclient is executed.

    This way I know the action has been completed, and I found the code easier for others to follow

  • You can set the HttpClient to work Synchronously. On the open() method you can set it to sync passing a third parameter as false.

    Ti.Network.createHTTPClient.open("POST", http://myserver.com, false);

    take a look in the official documentation: http://developer.appcelerator.com/apidoc/mobile/latest/Titanium.Network.HTTPClient.open-method.html
